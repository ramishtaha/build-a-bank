# ADR-0007: API Gateway on the MVC (servlet) stack; service-to-service via declarative HTTP interfaces

- **Status:** Accepted
- **Date:** 2026-06-10
- **Deciders:** Build-a-Bank (autonomous senior default)
- **Step:** 15 — API Gateway / BFF + service-to-service HTTP

## Context
Step 15 introduces a single front door (the gateway) and the first service-to-service HTTP call. Two choices,
both with a real compatibility/architecture dimension on the pinned Boot 4.0.6 / Spring Cloud 2025.1.1 set.

1. **Which Spring Cloud Gateway?** It ships in two flavours:
   - **Reactive** (`spring-cloud-starter-gateway-server-webflux`) — built on WebFlux/Netty (reactive stack).
   - **Servlet/MVC** (`spring-cloud-starter-gateway-server-webmvc`) — built on Spring MVC + a blocking HTTP client.
2. **How do services call each other?** `RestTemplate` (legacy), `RestClient` directly, OpenFeign (Spring Cloud),
   or a **declarative HTTP interface** (`@HttpExchange` + `HttpServiceProxyFactory`, Spring Framework 6.1+/7).

## Decision

### Gateway: the SERVLET (MVC) variant
Use **`spring-cloud-starter-gateway-server-webmvc`** (version managed by the Spring Cloud 2025.1.1 BOM; resolves to
the 5.0.x line). Rationale: the whole course is **Spring MVC + virtual threads** (the Operating Contract calls WebFlux
"an advanced aside"). The reactive gateway would drag WebFlux/Netty and a different programming model into the stack
for no benefit here. The MVC gateway fits the existing model, is configured with the same `application.yml` style, and
its blocking forwarding is fine with virtual threads.

- Config prefix is **`spring.cloud.gateway.server.webmvc.routes`** (the 2025 rename; `spring.cloud.gateway.mvc.routes`
  is deprecated). Routes use a `Path` predicate + `StripPrefix` + filters (e.g. `AddResponseHeader`).
- Routing scheme: a **service prefix** (`/cif/**`, `/bank/**`) stripped before forwarding, so external
  `/cif/api/customers/1` reaches CIF's own `/api/customers/1`.

**Verified:** `spring-cloud-starter-gateway-server-webmvc` resolves on the BOM and an integration test boots the
gateway and forwards a request to an in-test stub, with `StripPrefix` and `AddResponseHeader` working
(`gateway/GatewayRoutingTest`).

### Service-to-service: declarative HTTP interfaces (`@HttpExchange`)
Use Spring's **declarative HTTP interface** — an `@HttpExchange` interface whose implementation is generated by
`HttpServiceProxyFactory` over a `RestClient`. Rationale: type-safe, no hand-written HTTP plumbing, no extra
dependency (it's in Spring Framework), and the modern successor to `RestTemplate`/OpenFeign for in-Spring calls.
The `RestClient` is built with explicit **connect and read timeouts** (a service call must fail fast, never hang on a
slow dependency). Proper circuit-breaking/retry with **Resilience4j** is Step 37; here we keep the resilience to
timeouts and note the rest.

**Verified:** `CifClient` (`@GetExchange`) deserializes a customer and trips its read timeout on a slow stub
(`demand-account/CifClientTest`).

## Consequences
- ✅ One MVC stack across the platform — no WebFlux split; gateway config matches the services' style.
- ✅ Type-safe, dependency-free service-to-service calls with enforced timeouts (resilience baseline).
- ✅ No new *direct* deps for the client (Spring Framework); the gateway dep is BOM-managed.
- ⚠️ The MVC gateway is blocking; under high fan-out it relies on virtual threads / pool sizing (Steps 11, 37) to
  scale — acceptable and intentional here.
- ⚠️ Timeouts alone aren't full resilience — no circuit breaker, bulkhead, or retry-with-backoff yet. **Resilience4j
  in Step 37**; load balancing / discovery later. Flagged, not hidden.
- 🔁 The gateway is where cross-cutting edge concerns (auth at the edge, rate limiting, correlation-id origination)
  will live — Spring Security (Steps 16–17), rate limiting (Step 51).
