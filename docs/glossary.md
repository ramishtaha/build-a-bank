## Step 11 — Concurrency & Thread Safety

- **JMM (Java Memory Model)** — the JLS contract defining what one thread is guaranteed to see of another's actions. It governs three things: atomicity, visibility, and ordering.
- **Atomicity** — whether an operation happens "all at once" or can be interrupted partway by another thread. `balance += amount` is *not* atomic (it's read-modify-write).
- **Visibility** — whether a write by one thread is guaranteed to be *seen* by another. Without coordination, a write may sit in a CPU cache/register and never (or only late) become visible.
- **Ordering** — the compiler/JIT/CPU may reorder instructions for speed; another thread can observe a surprising order unless a happens-before edge forbids it.
- **happens-before** — the JMM relation that, when it holds between actions A and B, guarantees A's writes are visible to B and A is ordered before B. Created by lock unlock→lock, `volatile` write→read, `start()`, `join()`, and final-field publication.
- **race condition** — a bug whose outcome depends on the unpredictable timing/interleaving of threads.
- **lost update** — a specific race where two read-modify-write operations interleave so one update overwrites (and discards) the other.
- **`synchronized` / monitor** — Java's intrinsic lock. Entering a `synchronized` method/block acquires the object's monitor (mutual exclusion); the release→acquire pair creates a happens-before edge (visibility + ordering).
- **`volatile`** — a field modifier giving visibility + ordering (a memory barrier) and atomic single read/write, but **not** atomicity of compound actions. For flags, not counters.
- **CAS (compare-and-swap)** — a single atomic CPU instruction: "set this location to *new* only if it currently equals *expected*." The basis of lock-free algorithms; retries on conflict instead of blocking.
- **`AtomicLong` / `AtomicInteger`** — a single `volatile` value updated via a CAS loop (`addAndGet`, `incrementAndGet`). Lock-free, ideal at low/medium contention.
- **`LongAdder`** — a contention-friendly counter that stripes its value across multiple `Cell`s (threads usually hit different cells), summed on read via `sum()`. Beats `AtomicLong` under heavy write contention.
- **virtual thread** — a lightweight JVM-scheduled thread (stable Java 21) that runs many-to-few on platform carrier threads; cheap to create by the million.
- **carrier thread** — a platform (OS) thread that actually runs virtual threads; a virtual thread **mounts** a carrier to run and **unmounts** it when it blocks.
- **mount / unmount** — a virtual thread mounts onto a carrier while running and unmounts (freeing the carrier) when it blocks, which is why blocking is cheap.
- **`ExecutorService`** — a task-submission abstraction over a thread pool; `AutoCloseable` since Java 19 (its `close()` performs an orderly shutdown that *waits* for submitted tasks).
- **`CompletableFuture`** — composable asynchronous results (`thenCombine`, `thenCompose`, `allOf`) without manual thread juggling.
- **`Semaphore`** — a counting permit holder; `Semaphore(n)` lets at most `n` threads into a section at once — a clean way to bound concurrency.
- **`CyclicBarrier`** — a sync point where N threads `await()` until all arrive, then all proceed; used here to *force* a deterministic lost-update interleaving.
- **deadlock** — two+ threads each holding a lock the other needs, so none can proceed. Fixed by a global lock ordering or `tryLock` with timeout.
- **double-checked locking** — the lazy-singleton trick that is broken unless the field is `volatile` (reordering can publish a half-constructed object).
- **false sharing** — two unrelated fields on the same 64-byte cache line, causing cores to fight over the line; mitigated by `LongAdder` / `@Contended`.
- **TOCTOU (time-of-check-to-time-of-use)** — a race where state changes between checking it and acting on it (e.g. check balance, then debit); a security bug, not just a correctness one. Fix by making check-and-act atomic.

## Step 12 — Demand Account, the Double-Entry Ledger & Transactions

- **ACID** — the four guarantees of a database transaction: **A**tomicity (all-or-nothing), **C**onsistency (invariants hold), **I**solation (concurrent transactions don't corrupt each other), **D**urability (committed survives a crash).
- **double-entry bookkeeping** — every money movement is recorded as two entries of equal amount: a **DEBIT** on the payer and a **CREDIT** on the payee, sharing one `transactionId`. The entry log is append-only and always nets to zero, so the books always balance.
- **ledger / `LedgerEntry`** — the append-only table of entries (one leg per row). Never updated or deleted — an immutable audit trail.
- **materialized balance** — a `balance` column kept on the account and updated inside the transfer transaction, so reads are O(1); contrasted with a **derived balance** (summing the ledger on each read, O(entries)).
- **`@Transactional`** — Spring's declarative transaction boundary. A proxy opens a transaction before the method, commits on normal return, rolls back on a (runtime) exception.
- **propagation** — what happens when a transactional method calls another: `REQUIRED` (join or start), `REQUIRES_NEW` (suspend caller, run independently), `NESTED` (savepoint), plus `SUPPORTS`/`MANDATORY`/`NEVER`/`NOT_SUPPORTED`.
- **`REQUIRES_NEW`** — suspends the caller's transaction and runs in a brand-new one that commits independently; used for audit rows that must survive an outer rollback. Only takes effect across a bean boundary.
- **rollback rules** — by default `@Transactional` rolls back on `RuntimeException`/`Error` only; checked exceptions **commit** unless `rollbackFor` is set.
- **`readOnly = true`** — a transaction hint that no writes happen; Hibernate sets the flush mode to manual (skips dirty-checking) and the DB/driver can optimize.
- **self-invocation pitfall** — calling a `@Transactional` (or `@Async`/`@PreAuthorize`) method via `this.` inside the same bean bypasses the Spring proxy, so the advice never runs (Step 7).
- **optimistic locking (`@Version`)** — no lock during the read; an `@Version` column makes the `UPDATE` carry `WHERE version = ?` and `SET version = version + 1`. A stale write matches 0 rows → `ObjectOptimisticLockingFailureException`. Detect-and-retry; best for rare conflicts.
- **`ObjectOptimisticLockingFailureException`** — Spring's exception thrown when an optimistic-lock version check fails (a concurrent write bumped the version first).
- **pessimistic locking (`SELECT … FOR UPDATE`)** — take a row write-lock at read time so other transactions **block** until commit. Expressed in Spring Data as `@Lock(LockModeType.PESSIMISTIC_WRITE)`. Lock-and-wait; best for high contention.
- **`@Lock(LockModeType.PESSIMISTIC_WRITE)`** — the Spring Data annotation that makes Hibernate emit `SELECT … FOR UPDATE` for a query method.
- **lock ordering** — acquiring multiple locks in a consistent global order (here, by account number) so concurrent operations can't form a hold-and-wait deadlock cycle (Step 11).
- **`40P01` / deadlock detection** — Postgres's error code when it detects a deadlock and aborts one transaction to break it; lock ordering avoids it entirely.
- **lost update** — two read-modify-writes interleave so one overwrites and discards the other; the failure mode the transfer lock prevents (also Step 11).
- **double-spend / TOCTOU** — spending the same balance twice via the check-then-debit race; the security framing of the lost-update bug (Step 11).
- **`BigDecimal` / minor units** — exact decimal money (`numeric(19,4)`), compared with `compareTo` (scale-insensitive), never `double`/`float`. "Minor units" = the smallest currency unit (cents) as a concept.
- **bulk `@Modifying` update** — a JPQL `UPDATE`/`DELETE` that runs directly against the table, bypassing the persistence context and `@Version` (no dirty-checking, no optimistic check) — which is why `applyBalanceUnsafe` can demonstrate a true lost update.
- **`@ServiceConnection`** — a Spring Boot 3.1+ test annotation that wires the app's `DataSource` to a running Testcontainers container automatically (no JDBC URL/credentials in test config).
- **`@Enumerated(EnumType.STRING)`** — persists an enum as its stable text name (e.g. `"DEBIT"`) rather than its ordinal, so reordering the enum can't corrupt stored rows.

## Step 13 — Spring MVC / REST Deep (Problem Details, OpenAPI, Filters & Interceptors)

- **`DispatcherServlet`** — the Spring MVC **front controller**: the single servlet (mapped to `/`) that receives every request and orchestrates handler mapping, argument binding, invocation, return-value handling, and exception resolution.
- **`HandlerMapping`** — the component that matches a request (path + method + headers + content type) to a handler. `RequestMappingHandlerMapping` matches `@RequestMapping`/`@GetMapping`/… and returns a `HandlerExecutionChain` (handler + interceptors).
- **`HandlerAdapter`** — the component that actually invokes the matched handler. `RequestMappingHandlerAdapter` resolves method arguments (`@PathVariable`, `@RequestParam`, `@RequestBody`, …), runs `@Valid`, calls your method, and processes the return value.
- **`HttpMessageConverter`** — converts between HTTP bodies and Java objects. The Jackson converter deserializes `@RequestBody` JSON into your record and serializes the returned DTO back to JSON.
- **content negotiation** — choosing the response representation/converter from the request's `Accept` header (and the handler's producible types). JSON by default here; errors negotiate to `application/problem+json`.
- **`@RestController`** — `@Controller` + `@ResponseBody`: a web controller whose return values are written straight to the response body (serialized by a message converter), not resolved to a view.
- **`@ControllerAdvice` / `@RestControllerAdvice`** — a global, cross-controller bean of `@ExceptionHandler` (and other) methods. `@RestControllerAdvice` adds `@ResponseBody` so handlers' return values become the response body. Spring's `ExceptionHandlerExceptionResolver` picks the most specific handler for a thrown exception.
- **`ResponseEntityExceptionHandler`** — the Spring base class for a `@ControllerAdvice` that already turns the framework's **built-in** MVC exceptions (validation, unreadable body, 404/405) into `ProblemDetail`. Extend it and override hooks (e.g. `handleMethodArgumentNotValid`) to enrich them.
- **`ProblemDetail`** — `org.springframework.http.ProblemDetail` (Spring 6+): the object representing an RFC 9457 error. Returning it from an `@ExceptionHandler` sets the HTTP status from it and serializes `application/problem+json`.
- **RFC 9457 / `application/problem+json`** — the standard "Problem Details for HTTP APIs" media type and body shape: `type` (problem-kind URI), `title`, `status`, `detail`, `instance`, plus arbitrary **extension members** (we add `errors`). Successor to RFC 7807.
- **extension member** — a custom field added to a Problem Detail beyond the five standard ones, via `problem.setProperty("errors", map)`; RFC 9457 explicitly allows them.
- **`MethodArgumentNotValidException`** — the exception thrown when `@Valid @RequestBody` binding fails Bean Validation; `ResponseEntityExceptionHandler` routes it to `handleMethodArgumentNotValid`, where we attach the per-field `errors` map.
- **422 Unprocessable Entity** — the status for a **well-formed** request that can't be fulfilled (here, an overdraw), as opposed to **400 Bad Request** for a malformed/invalid request.
- **`Filter` (`jakarta.servlet.Filter`)** — a **servlet-container-level** interceptor that wraps the entire `DispatcherServlet`; sees every request (even 404s with no handler) and has no handler context. A `Filter` bean is **auto-registered** by Boot.
- **`OncePerRequestFilter`** — Spring's base `Filter` guaranteeing the body runs **exactly once per request** (dedupes async dispatches/forwards); you implement `doFilterInternal`.
- **`HandlerInterceptor`** — a **Spring-MVC-level** interceptor that runs *around the matched handler* with three hooks: `preHandle` (before; `false` short-circuits), `postHandle` (after the handler, before commit), `afterCompletion` (always, even on exception). Must be **registered** via `WebMvcConfigurer.addInterceptors(...)`.
- **`WebMvcConfigurer`** — the callback interface for customizing Spring MVC (e.g. `addInterceptors`, CORS, formatters); all methods are `default`, so you override only what you need.
- **correlation id (`X-Request-Id`)** — a per-request identifier stamped on the response (reused from inbound or minted) so one request can be traced across logs/services; set by the `RequestIdFilter` and wired into tracing in Step 36.
- **request attribute** — per-request state stored on the `HttpServletRequest` (`setAttribute`/`getAttribute`) rather than on a shared singleton component — the thread-safe place for an interceptor's start time.
- **OpenAPI / Swagger UI** — OpenAPI is the machine-readable API spec; **Swagger UI** is a browsable HTML client rendered from it. springdoc serves the spec at `/v3/api-docs` (OpenAPI 3.1) and the UI at `/swagger-ui.html`.
- **springdoc-openapi** — the maintained library that generates the OpenAPI spec + Swagger UI from your controllers/DTOs/validation annotations. The `-webmvc-ui` starter targets the servlet stack; **3.0.x** supports Boot 4 (2.8.x targets Boot 3). Replaces the dead **springfox**.
- **`MockMvc` / `@WebMvcTest`** — a web-layer **slice** test: loads only the controller + advice + MVC infra (filters, interceptors, Jackson) with the service mocked (`@MockitoBean`); fast, no DB. Contrast with `@SpringBootTest(RANDOM_PORT)`, which boots the whole app and sends real HTTP.

## Step 14 — API Design, Versioning, Idempotency & Webhooks

- **API versioning** — letting clients pin to a compatible API shape as it evolves. Three common strategies: **URI** (`/api/v1/...` — visible, cacheable, curl-able; chosen here), **header** (`Accept: application/vnd.bank.v1+json` or `X-API-Version` — clean URLs but invisible in logs), and **media-type** (most RESTful, heaviest). New versions are added **additively**, never by mutating the old one.
- **`Deprecation` / `Sunset` / `Link` headers (RFC 8594)** — the standard, machine-readable way to retire an endpoint gracefully: `Deprecation: true` (it's deprecated), `Sunset: <HTTP-date>` (when it'll be removed), `Link: <successor-uri>; rel="successor-version"` (the replacement). The old endpoint keeps working until the sunset date — never a flag-day break.
- **idempotency** — the property that performing an operation twice has the same effect as once. `GET`/`PUT`/`DELETE` are naturally idempotent; `POST` (create a transfer) is **not** — a retry creates a second resource unless made idempotent.
- **`Idempotency-Key`** — a client-supplied, per-logical-operation key (a UUID) sent as an HTTP header so the server can dedupe retries. The server stores `key → result` and returns the stored result on any retry with the same key (Stripe's model).
- **idempotency key store** — the table (here `idempotency_key`, PK = the key) that remembers `key → transactionId`. Its **PRIMARY KEY uniqueness is the concurrency guard**: two racing duplicates can't both insert, so only one transfer commits.
- **HMAC-SHA256** — a **H**ash-based **M**essage **A**uthentication **C**ode: a keyed hash (`javax.crypto.Mac`) over the message using a shared secret. Only a secret-holder can produce a signature matching a given payload, so it proves **authenticity** (it came from us) and **integrity** (the body wasn't altered). Immune to length-extension attacks (unlike a naive `hash(secret∥body)`).
- **signed material** — exactly the bytes the signature is computed over. Here it's `"<timestamp>.<body>"` (UTF-8): binding the timestamp into the hash is what makes the replay window tamper-proof.
- **replay protection** — rejecting a captured-but-still-valid request that's re-sent later. Achieved by signing a **timestamp** into the payload and rejecting on verify any timestamp outside a **tolerance window** (e.g. ±300s), even if the signature is correct.
- **constant-time comparison** — comparing two byte arrays in time independent of where the first mismatch is (`MessageDigest.isEqual`), so an attacker can't use response timing to brute-force a signature byte by byte. Never compare signatures with `String.equals`/`==`.
- **at-least-once delivery** — a delivery guarantee where a message may arrive **one or more** times (because the sender retries on failure), but never zero. Webhook delivery here is at-least-once.
- **idempotent receiver** — a webhook consumer that dedupes repeated events (by event id) so at-least-once delivery doesn't cause double processing. The receiver's responsibility, and the reason senders can safely retry.
- **dual-write problem** — the inconsistency that arises when one operation must update two systems (here: commit the DB transaction **and** deliver the webhook) without a shared transaction. If one succeeds and the other fails, they disagree. Sending the webhook *after* commit leaves the "committed but not delivered" gap.
- **Outbox pattern** — the fix for the dual-write problem (Step 20): write the event into an `outbox` table in the **same transaction** as the business change, then a separate process delivers it and marks it sent. The event is durable iff the transaction committed.
- **`Pageable`** — Spring Data's pagination *request* object (which page, what size, what sort). Bound from `?page=&size=&sort=field,dir` by `PageableHandlerMethodArgumentResolver`; defaults supplied via `@PageableDefault`.
- **`Page<T>`** — Spring Data's pagination *result*: the slice of rows plus metadata (`totalElements`, `totalPages`, `number`, `size`). A `Page<>` repository method runs a windowed `SELECT … LIMIT/OFFSET` plus a `COUNT` query.
- **`PageResponse` (stable envelope)** — a custom DTO record (`content`, `page`, `size`, `totalElements`, `totalPages`) the API owns and serializes, **instead of** Spring Data's `Page` (whose JSON shape is an internal detail Spring warns against exposing).
- **offset pagination** — `page`/`size` (`LIMIT/OFFSET`): simple, supports random page access, but degrades at depth and can skip/duplicate rows under concurrent writes.
- **cursor (keyset) pagination** — "give me N rows after id X / createdAt T": stable and fast at any depth, but no random page access. Preferred for high-volume ledgers.
- **`@RequestHeader`** — binds an HTTP request header to a controller method parameter; `required = false` makes the header optional (the parameter is `null` when absent) — how the optional `Idempotency-Key` is read.
- **`@PageableDefault`** — sets the default page number/size/sort for an injected `Pageable` when the client omits the query params; also a cheap cap on unbounded responses.
- **Jackson 3 vs Jackson 2 (Spring Boot 4)** — Boot 4's web stack defaults to **Jackson 3** (`tools.jackson`), so a Jackson-2 (`com.fasterxml.jackson.databind`) `ObjectMapper` **bean** isn't auto-created. Injecting one fails at context startup; own a `new ObjectMapper()` or use the Jackson-3 mapper.

## Step 15 — API Gateway / BFF & Service-to-Service HTTP

- **API Gateway** — a single service in front of all the others that receives every external request and **routes** it to the right internal service. The natural home for **edge cross-cutting concerns** (routing, auth, rate limiting, TLS termination, correlation ids, request/response shaping). Clients see one address; internal services stay private. Business logic does **not** belong here.
- **BFF (Backend-For-Frontend)** — a gateway **specialized per client type** (a web BFF, a mobile BFF) that aggregates/tailors responses for that frontend (e.g. combine three service calls into one trimmed payload for a phone). Trade-off: more gateways to operate, each simpler and evolving with its client.
- **route** — a gateway rule: an `id`, a target `uri`, **predicates** (when it matches), and **filters** (transforms). The first matching route wins.
- **predicate** — a route's match condition (e.g. `Path=/cif/**` matches any path starting with `/cif/`; the `**` is an Ant-style wildcard for the remaining segments). Also available: method, header, host predicates.
- **filter** — a transform applied to the request and/or response of a matched route (rewrite path, add/remove headers, rate-limit, authenticate). Runs in declared order.
- **`StripPrefix=N`** — a gateway filter that removes the first **N** path segments before forwarding, so the downstream receives its own path (`/cif/api/customers/1` with `StripPrefix=1` → `/api/customers/1`).
- **`AddResponseHeader` / `AddRequestHeader`** — gateway filters that tag the response (or forwarded request) with a header — here `X-Gateway: build-a-bank` proves a filter ran; later this slot carries correlation/auth headers.
- **Spring Cloud Gateway Server WebMVC** — the **servlet** (Spring MVC) gateway (`spring-cloud-starter-gateway-server-webmvc`), as opposed to the **reactive** one (`-server-webflux`, WebFlux/Netty). We use the servlet variant to stay on the MVC + virtual-threads stack; its blocking forwards scale cheaply on virtual threads. Config prefix: **`spring.cloud.gateway.server.webmvc.routes`** (the 2025 rename; `spring.cloud.gateway.mvc.routes` is deprecated and won't bind).
- **declarative HTTP interface (`@HttpExchange`)** — an annotated **interface** describing remote calls (`@HttpExchange` base path + `@GetExchange`/`@PostExchange` methods, `@PathVariable`/`@RequestParam`/`@RequestBody` bindings). You declare *what*; Spring generates the *how*. The modern, type-safe successor to hand-written `RestTemplate` calls and to OpenFeign for in-Spring use (Spring Framework 6.1+).
- **`@GetExchange`** — the declarative-client analogue of `@GetMapping`: declares a `GET` to the given path (combined with the interface's `@HttpExchange` base path).
- **`HttpServiceProxyFactory`** — the factory that reads a `@HttpExchange` interface and builds a **dynamic proxy** implementing it, turning each method call into an HTTP request via a backing client adapter.
- **`RestClientAdapter`** — the bridge that lets `HttpServiceProxyFactory` drive its calls through a `RestClient` (`RestClientAdapter.create(restClient)`).
- **`RestClient`** — Spring Framework 6.1+'s fluent, **synchronous** HTTP client (the modern `RestTemplate` replacement). Built with a base URL and a request factory; its message converters (de)serialize bodies.
- **`JdkClientHttpRequestFactory`** — Spring's request factory wrapping the JDK's `java.net.http.HttpClient`; where you set the **read timeout** (`setReadTimeout`) for a `RestClient`.
- **connect timeout** — how long to wait to **establish** the TCP/TLS connection before giving up (set on the JDK `HttpClient`).
- **read timeout** — how long to wait for the **response** after the request is sent (set on `JdkClientHttpRequestFactory`). The one that saves you from a connected-but-slow downstream.
- **`ResourceAccessException`** — Spring's "transport-level problem" exception. A read-timeout (`HttpTimeoutException`) surfaces as this — a clean, catchable failure instead of a hung thread.
- **cascading failure** — when one slow/failed component, called without a timeout, ties up callers' threads until they're exhausted, propagating the outage system-wide. Timeouts (then circuit breakers) are the availability control against it.
- **circuit breaker** — (Step 37, Resilience4j) a resilience pattern that "opens" after a failure threshold and fails fast (or serves a fallback) without even trying, protecting both caller and callee and giving the downstream room to recover. Pairs with bulkheads and retries-with-backoff.
- **temporal coupling** — the property of synchronous calls that the caller is affected *now* if the callee is down/slow (vs async messaging, which decouples them in time).
- **`@DynamicPropertySource`** — a JUnit/Spring test hook that runs **before** the context starts, used to feed a runtime-discovered value (here a random stub-server port; in Step 8, a Testcontainers JDBC port) into Spring config before beans/routes are wired.

## Step 16 — Spring Security Deep I (Filter Chain, JWT & Password Encoding)

- **security filter chain** — the ordered list of servlet filters Spring Security places in front of your app; every request runs it *before* the `DispatcherServlet`/controller. It authenticates (else 401) then authorizes (else 403). Registered as one `FilterChainProxy` that delegates to your `SecurityFilterChain` bean(s).
- **`SecurityFilterChain` (bean)** — the modern way to configure Spring Security: a `@Bean` built from `HttpSecurity` via the lambda DSL (`http.csrf(...).sessionManagement(...).authorizeHttpRequests(...).oauth2ResourceServer(...).build()`). Replaces the **removed** `WebSecurityConfigurerAdapter` (gone in Spring Security 6).
- **authentication (401)** — establishing *who you are* from the request (here: decoding+validating the Bearer JWT). Missing/invalid identity → **401 Unauthorized** (via the `AuthenticationEntryPoint`).
- **authorization (403)** — deciding *whether the established identity is allowed* to do this (e.g. needs `ROLE_ADMIN`). Authenticated but not permitted → **403 Forbidden** (via the `AccessDeniedHandler`).
- **`authorizeHttpRequests`** — the DSL block declaring per-path access rules, evaluated **top-down, first match wins**: `permitAll()` (public), `authenticated()` (any valid identity), `hasRole("ADMIN")` (role-restricted). The successor to the older `authorizeRequests`.
- **`requestMatchers(...)`** — selects the path(s) a rule applies to (e.g. `/api/auth/login`, `/api/auth/admin`). The successor to `antMatchers` (removed in Security 6).
- **`hasRole("ADMIN")`** — requires the authority **`ROLE_ADMIN`** (Spring adds the `ROLE_` prefix for you in this check). Contrast `hasAuthority("ROLE_ADMIN")`, which expects the full string.
- **JWT (JSON Web Token)** — a compact, signed token of three base64url parts: **`header.claims.signature`**. The header names the algorithm; the claims carry identity/metadata; the signature proves integrity/authenticity. **Signed, not encrypted** — the payload is readable by anyone (don't put secrets/PII in it).
- **claims** — the JWT payload fields: `sub` (subject = username), `roles` (custom — the authorities), `iss` (issuer), `iat` (issued-at), `exp` (expiry). The decoder enforces `exp`; we map `roles` → authorities.
- **OAuth2 resource server** — a service that **validates** incoming Bearer JWTs (it doesn't issue them). Spring's `oauth2ResourceServer(jwt(...))` installs the `BearerTokenAuthenticationFilter` which decodes+validates the token and sets the `Authentication`.
- **`BearerTokenAuthenticationFilter`** — the filter that extracts `Authorization: Bearer <jwt>`, hands it to the `JwtDecoder`, and on success populates the `SecurityContext`.
- **`JwtEncoder` / `NimbusJwtEncoder`** — issues (signs) JWTs. `new NimbusJwtEncoder(new ImmutableSecret<>(secretKey))` signs with the shared HMAC secret over a `JwtClaimsSet` + `JwsHeader`.
- **`JwtDecoder` / `NimbusJwtDecoder`** — validates JWTs: `NimbusJwtDecoder.withSecretKey(key).macAlgorithm(HS256).build()` checks the **signature** and **expiry**, with the algorithm **pinned** (so a token can't dictate `alg`).
- **`JwtAuthenticationConverter` / `JwtGrantedAuthoritiesConverter`** — maps JWT claims to Spring authorities. We set `authoritiesClaimName("roles")` and an empty `authorityPrefix` (the claim already carries `ROLE_`), so `roles:["ROLE_USER"]` → authority `ROLE_USER`.
- **HMAC (HS256)** — symmetric JWT signing: **one shared secret** signs *and* validates. Simple for one issuer+validator, but every validator can also **forge**. Needs a **≥ 256-bit (32-byte)** key or it fails at startup.
- **asymmetric signing / JWKS** — the issuer signs with a **private** key; validators use the **public** key (fetched from a **JWKS** endpoint, `/.well-known/jwks.json`) and *cannot* mint tokens. The move when many independent services validate (Step 17/41).
- **BCrypt / `BCryptPasswordEncoder`** — a deliberately **slow, salted, one-way** password hash. `encode` emits `$2a$10$<22-char-salt><31-char-hash>` (cost + salt baked in); `matches(raw, hash)` re-derives and constant-time-compares. Slow defeats brute force; per-hash salt defeats rainbow tables.
- **salt** — random bytes mixed into a password before hashing so identical passwords produce **different** hashes (no precomputed/rainbow-table attacks). BCrypt embeds a fresh 16-byte salt in each hash string.
- **cost factor (work factor)** — BCrypt's tuning knob: the hash runs `2^cost` rounds (default 10). Higher = slower = harder to brute-force; raise it as hardware improves.
- **`PasswordEncoder.matches`** — verify a raw attempt against a stored hash (constant-time). **Never** compare passwords with `equals`, and never store/log plaintext.
- **CSRF (Cross-Site Request Forgery)** — an attack that tricks a browser into sending a request with the user's **ambient** credentials (cookies/session sent automatically). Disabled here because a **stateless Bearer-token** API has no ambient credentials to ride; re-enable for cookie/session flows.
- **`SessionCreationPolicy.STATELESS`** — tells Spring Security **not** to create or use an `HttpSession`; identity is re-established from the token on every request (the stateless model that scales horizontally).
- **security headers** — safe defaults Spring Security sets on responses, e.g. `X-Content-Type-Options: nosniff` (don't MIME-sniff), `X-Frame-Options` (clickjacking), `Cache-Control`. We assert `nosniff` is present.
- **`SecurityContextHolder`** — the holder of the current `Authentication`, backed by a **`ThreadLocal`** (per request, cleared at the end). Why a controller can take an `Authentication` parameter and see only *its own* request's identity.
- **`FACTOR_BEARER`** — a Spring Security **7** *authentication-factor* authority granted alongside roles to record *how* you authenticated (here: a bearer token). An internal marker for step-up/MFA reasoning (Step 17), not a role — we filter it out of the `/me` response so the API contract reports only `ROLE_*`.

## Step 17 — Spring Security Deep II (Resource Servers, RS256/JWKS, Method Security & Modern Auth)

- **asymmetric signing (RS256)** — JWT signing with an RSA **key pair**: the issuer signs with the **private** key, verifiers validate with the **public** key. `RS256` = RSASSA-PKCS1-v1_5 over SHA-256 (declared as the JWS header `alg`). Lets many services verify without being able to **forge** — the multi-service default. Contrast **HMAC (HS256)**, one shared secret that both signs and validates.
- **JWKS (JSON Web Key Set)** — the issuer's **public** signing keys published as JSON at a URL (here `GET /oauth2/jwks`), shaped `{"keys":[{"kty":"RSA","kid":...,"n":...,"e":"AQAB"}]}`. Contains **public material only** (`kty/kid/n/e`) — never the private exponent `d` or primes `p`/`q`. Resource servers fetch it to validate signatures. (RFC 7517.)
- **`kid` (key id)** — an identifier carried in both the JWKS entry and each token's JWS header, so a verifier with several keys picks the right one. The hook that makes **key rotation** seamless.
- **key rotation** — replacing a signing key without downtime: publish the new key in the JWKS alongside the old, sign new tokens with the new `kid`, and retire the old key once all tokens signed with it have expired. Verifiers refresh the JWKS on an unknown `kid`.
- **OAuth2 resource server** — a service that protects endpoints by validating a **Bearer access token** (signature + expiry + claims) on each request, statelessly — no login UI, no session. Configured with `oauth2ResourceServer(jwt(...))`. Distinct from an **authorization server** (auth), which *issues* tokens.
- **`jwk-set-uri`** — the Spring property (`spring.security.oauth2.resourceserver.jwt.jwk-set-uri`) pointing a resource server at the issuer's JWKS. Boot builds a `JwtDecoder` that fetches keys **lazily** (on the first token, so the service boots even if the issuer is down) and **caches** them.
- **`JwtEncoder` / `NimbusJwtEncoder`** — signs JWTs. Backed by a `JWKSource` holding the **private** key; `JwsHeader.with(SignatureAlgorithm.RS256)` declares the algorithm.
- **`JwtDecoder` / `NimbusJwtDecoder`** — validates JWTs: signature (against the public key, via `withPublicKey(...)` or `jwk-set-uri`) + standard clock claims (`exp`/`nbf`). Builds the `Authentication`.
- **`BearerTokenAuthenticationFilter`** — the resource-server filter that extracts `Authorization: Bearer <jwt>`, runs the `JwtDecoder`, and stores the resulting `JwtAuthenticationToken`. No token / invalid → **401** via the `BearerTokenAuthenticationEntryPoint`.
- **`JWKSource` / `ImmutableJWKSet`** — Nimbus abstraction for "where the encoder finds keys to sign with." `ImmutableJWKSet<>(new JWKSet(rsaKey))` wraps the full key (private included) — used **only** to sign, never serialized to clients.
- **`RSAKey` / `RSAKeyGenerator`** — Nimbus types for an RSA JWK. `new RSAKeyGenerator(2048).keyID(...).generate()` creates a 2048-bit key with a `kid`; `.toPublicJWK()` strips private material for publishing; `.toRSAPublicKey()` / `.toPrivateKey()` extract the JCA halves.
- **method security (`@EnableMethodSecurity` / `@PreAuthorize`)** — fine-grained, domain-level authorization expressed on a method via SpEL (`hasRole('ADMIN')`), enforced by an AOP interceptor **before** the method runs (`AccessDeniedException` → 403). `@EnableMethodSecurity` registers the infrastructure; without it, `@PreAuthorize` is silently inert. Applies to **direct in-process calls too**, not just HTTP (the blast-radius caveat).
- **URL authorization vs method security** — URL rules (`authorizeHttpRequests`) gate the HTTP **edge** coarsely; method security expresses **reusable, fine-grained** rules on methods. Use both; mind that method security on a service method affects unit tests that call it directly.
- **`@EnableWebSecurity`** — imports `HttpSecurityConfiguration` so an `HttpSecurity` bean exists. Implicit in a full app, but **required on the config** for a `@WebMvcTest` slice to start (else *"No qualifying bean of type HttpSecurity"*).
- **`jwt()` request post-processor** — `spring-security-test`'s `SecurityMockMvcRequestPostProcessors.jwt()`: injects a ready-made `JwtAuthenticationToken` (with chosen authorities) into a MockMvc request — tests **authorization** without real token decoding. Pairs with a `@MockitoBean JwtDecoder` to satisfy bean wiring in the slice.
- **identity propagation** — passing the caller's identity downstream: the gateway forwards the `Authorization` header so a token presented at the edge reaches each service, which validates it independently. (Hardening: strip client-supplied headers the gateway sets itself; full propagation + mTLS is Step 41/43.)
- **token revocation** — invalidating a token before its `exp`. Hard for self-contained JWTs (no per-request lookup); mitigate with short access-token lifetimes + revocable **refresh tokens**, a `jti` denylist, or key rotation. Opaque tokens + introspection give instant revocation at the cost of a network hop.
- **refresh token** — a longer-lived, revocable credential used to mint fresh short-lived access tokens without re-login (the frontend flow, Step 32). Revoking it cuts off future access-token minting.
- **MFA (multi-factor authentication)** — requiring more than one factor: something you *know* (password), *have* (device/OTP), or *are* (biometric).
- **passkey / WebAuthn** — a phishing-resistant, **device-bound public-key** credential: the authenticator (Face ID/fingerprint/security key) signs a server challenge with a private key that never leaves the device; the server verifies with the stored public key. No shared secret, nothing phishable. Spring has a `webAuthn()` DSL; the ceremony is browser-driven (Phase F).
- **step-up authentication** — requiring a **stronger/fresh** factor for a sensitive operation even when already authenticated (e.g. re-confirm before a large transfer). Spring Security 7 models factors as `FACTOR_*` authorities you can require with `@PreAuthorize`.
- **algorithm-confusion attack** — a JWT exploit where an attacker submits an `HS256` token signed with the RS256 **public** key (which is public) as the HMAC secret; a verifier that trusts the token's `alg` header validates it. Defend by pinning the expected algorithm to the **key**, not the token (Spring's JWKS decoder does this), and never accepting `alg: none`.

## Step 18 — Secure Coding & Threat Modeling (DevSecOps Shift-Left)

- **threat modeling** — a structured, repeatable way to ask "what can go wrong?" *before* an attacker does: model the system (DFD), enumerate threats (STRIDE), decide mitigations (ship or register), review on change. Shostack's four questions.
- **shift-left** — finding flaws earlier on the project timeline (design → code → test → deploy): a design-stage fix costs minutes, the same flaw in production costs a breach. Threat modeling is the design-stage security activity.
- **DFD (data-flow diagram)** — the threat model's map: **external entities** (clients, partners), **processes** (your services), **data stores** (databases), and the **flows** between them. Threats follow data, so you model flows — not deployment boxes.
- **trust boundary** — the line where data crosses between trust levels (internet→edge, edge→internal, service→DB, you→partner). Every crossing must **authenticate, authorize, and validate**; most findings live on a crossing where one check was assumed.
- **STRIDE** — six threat categories, each violating one property: **S**poofing (authentication), **T**ampering (integrity), **R**epudiation (non-repudiation), **I**nformation disclosure (confidentiality), **D**enial of service (availability), **E**levation of privilege (authorization).
- **STRIDE-per-element** — applying all six letters to *every* DFD element and flow, recording ✅ mitigated / 🟡 partial / 🔴 open with evidence. Breadth-first: you can't forget a *category* of bug.
- **attack tree** — goal-first depth: root = attacker goal ("move funds out"), branches = strategies, leaves = concrete techniques each marked closed/open. Complements STRIDE; our money tree showed every path closed *except* BOLA.
- **BOLA (Broken Object Level Authorization) / IDOR** — OWASP API1:2023, the #1 API risk: an endpoint acts on an object id from the request without checking the caller is entitled to *that object*. An **authorization** bug — authentication can pass. Fix: server-side ownership, enforced on every access.
- **object-level vs function-level vs property-level authorization** — *may you touch this object?* (API1 — needs an ownership model, schema-shaped) vs *may you call this operation?* (API5 — annotation-shaped, `@PreAuthorize`) vs *may you read/set this field?* (API3 — DTO-shaped). A system can pass two and fail the third — ours passed API5 and failed API1.
- **OWASP Top 10 (2021) / OWASP API Security Top 10 (2023)** — the standard ranked checklists of web-app and API risk categories (A01–A10 / API1–API10). Lenses for auditing real code, not compliance checkboxes — every row needs an evidence cell.
- **risk register** — the prioritized, owned output of a threat model: each open risk gets an **ID**, **severity** (likelihood × impact), mapped category, a *specified* fix, an **interim control**, and an **owner/ETA**. Records what you *won't* fix this sprint — exactly the items otherwise forgotten.
- **interim control** — what still stands while a risk is open (ours: BOLA requires an *authenticated* user — not anonymously exploitable). Recording it keeps the register honest in both directions.
- **secure-by-default** — the safe behavior requires no configuration; danger must be opted into explicitly (empty CORS allow-list ⇒ deny; headers always on). The inverse of "remember to lock it down in prod."
- **security headers** — response headers browsers obey: `X-Content-Type-Options: nosniff` (no MIME sniffing), `X-Frame-Options: DENY` (no framing), `Referrer-Policy: no-referrer` (no URL leak), HSTS. Written by Spring Security's `HeaderWriterFilter` on **every** response, including 401s.
- **clickjacking** — overlaying a victim's real (framed, invisible) UI under fake buttons so their click lands on "confirm transfer." Killed by `X-Frame-Options: DENY` (and CSP `frame-ancestors`).
- **MIME sniffing** — a browser guessing a response's content type instead of trusting `Content-Type`, which can turn a crafted "image" into executed HTML/JS. Disabled by `nosniff`.
- **HSTS (`Strict-Transport-Security`)** — tells the browser "only ever HTTPS for this host (and subdomains) for `max-age` seconds." Spring emits it **only over TLS** — its absence in plain-HTTP tests is correct, not a bug. (RFC 6797.)
- **CORS (Cross-Origin Resource Sharing)** — the browser mechanism by which a server opts *in* to cross-origin access. A **browser guardrail, not an access control**: non-browser clients ignore it; server-to-server calls carry no `Origin` so it never engages. The JWT stays the real gate.
- **preflight** — the browser's advance `OPTIONS` request (`Origin` + `Access-Control-Request-Method`) asking permission for a non-simple cross-origin call. Allowed → `Access-Control-Allow-Origin` (**ACAO**) in the response; denied → 403 with **no** ACAO (assert the *absence* — reflected origins are themselves a vuln). `Access-Control-Max-Age` lets the browser cache the verdict.
- **deny-by-default CORS** — an allow-list that starts **empty** (`setAllowedOrigins([])` from `app.security.cors.allowed-origins`): no browser origin may call unless explicitly listed (the React app arrives via one env var in Step 29).
- **`CorsConfigurationSource` (the two-beans gotcha)** — Spring Security finds the CORS policy by the bean **name** `corsConfigurationSource` via `.cors(Customizer.withDefaults())`. Injecting it *by type* into the chain fails: MVC's `mvcHandlerMappingIntrospector` also implements the interface → *"expected single matching bean but found 2."*
- **parameterized query / prepared statement** — SQL whose structure is parsed **before** values are bound (placeholders `?`); input is always *data*, never *code*, so injection is impossible by sequencing — the real fix for OWASP A03, vs fragile escaping/blocklists.
- **stacked query** — an injection payload that ends the statement and starts a destructive second one (`x'; DROP TABLE customer; --`). Bound as a parameter it's never parsed; our test proves the table survives.
- **mass assignment** — a client smuggling unexpected fields (`"balance": 9999999`) into a data bind. Guarded *by construction* when requests are explicit DTO records with only the intended fields (API3).
- **SSRF (Server-Side Request Forgery)** — tricking *your server* into calling an attacker-chosen URL (cloud metadata, internal services). Relevant to any outbound, configurable callback (our webhooks → R-004: host allow-list, block private ranges, egress proxy).
- **defense-in-depth** — independent, overlapping layers (headers *and* CORS *and* JWT *and* parameterization) so one control's failure isn't total compromise — while remembering which layer is the actual gate.
- **PASTA / LINDDUN** — heavier alternatives to STRIDE: PASTA is a seven-stage, business-risk-centric process; LINDDUN targets *privacy* threats (Linkability, Identifiability, …). STRIDE-per-element remains the best first tool.

## Step 19 — Distributed-Systems Theory & Trade-offs (CAP/PACELC, Consistency, Quorums, Clocks, Delivery)

- **8 fallacies of distributed computing** — the false assumptions single-machine intuition smuggles in: the network is reliable, latency is zero, bandwidth is infinite, the network is secure, topology doesn't change, there's one administrator, transport cost is zero, the network is homogeneous. Distributed-systems theory starts where these end.
- **(network) partition** — a failure where replicas can't communicate (cut links, dropped packets, a long GC pause that *looks* like death). Not optional to tolerate: it *will* happen.
- **CAP theorem** — during a partition, a distributed store keeps either **C**onsistency (linearizable reads) or **A**vailability (every request answered), never both (Brewer; proved by Gilbert & Lynch). Says nothing about the no-partition case.
- **CP / AP** — the two real CAP postures: **CP** refuses to serve during a partition rather than diverge (ZooKeeper, etcd — our lab throws `Unavailable`); **AP** keeps serving, accepts divergence, reconciles on heal (Dynamo-style).
- **"CA"** — not a real option for a distributed system: you can't opt out of partitions, only choose your behavior during one. A single-node DB is "CA" only by not being distributed.
- **PACELC** — Abadi's completion of CAP: **if P**artition → **A** or **C**; **E**lse → **L**atency or **C**onsistency. Even healthy replication costs round-trips: wait (EC) or ack-local-and-replicate-async (EL). Stores classify like PC/EC or PA/EL.
- **linearizability** — the strongest consistency model: as if there were one copy and every operation took effect atomically at a point in real time. CAP's "C". Costliest to provide.
- **sequential / causal / read-your-writes consistency** — the middle of the spectrum: one global order not tied to real time (sequential); cause→effect order preserved, concurrent ops reorderable (causal — what vector clocks implement); a session always sees its own writes (read-your-writes).
- **eventual consistency** — if writes stop, replicas converge. Cheap and available; until convergence, reads can be stale (our AP register mid-partition, our W=1/R=1 quorum).
- **consensus** — getting replicas to agree on a value despite failures (Paxos, Raft). Requires a **majority quorum** (`> N/2`) to make progress → tolerates ⌊(N−1)/2⌋ failures; safety is kept even when liveness stalls (see FLP).
- **quorum (`W + R > N`)** — with N replicas, write-set size W and read-set size R: strict inequality guarantees every read set intersects every write set (pigeonhole) → reads see the latest write; `W > N/2` additionally prevents two conflicting writes both committing. The consistency/latency **dial** of Dynamo-style stores.
- **strict vs sloppy quorum** — `W+R>N` (overlap guaranteed; strong reads) vs `W+R≤N` (a read quorum can miss the last write → stale reads, eventual consistency). Real Dynamo adds hinted handoff (stand-in replicas) and **read repair** (write the freshest version back to stale replicas during reads).
- **happens-before (`a → b`)** — Lamport's causality relation: program order within a process, plus send→receive across processes, transitively closed. Events related neither way are **concurrent** (`a ∥ b`).
- **Lamport clock** — one counter per process: tick on local events; `max(local, received) + 1` on receive. Guarantees `a → b ⇒ L(a) < L(b)`; the converse fails, so it **cannot detect concurrency**.
- **vector clock** — one counter per process, carried as a vector; componentwise max on receive, then bump own entry. `a → b` iff `a ≤ b` componentwise and `<` somewhere; incomparable = concurrent. Detects both order **and** conflicts, at O(N) space (Dynamo siblings, causal consistency).
- **delivery semantics** — what a channel can promise: **at-most-once** (may lose, never duplicates — fire-and-forget) or **at-least-once** (never loses, may duplicate — retries). A lost ack makes "exactly-once delivery" impossible to guarantee.
- **exactly-once effect** — the achievable substitute: tolerate at-least-once delivery and make processing **idempotent** so duplicates are no-ops. Kafka's "exactly-once" is this (idempotent producers + transactions), not magical delivery.
- **idempotent consumer** — a consumer that dedupes by a stable **message id** before applying state changes (`Set.add` returning false → skip; in production, a DB unique-constraint insert). Dedupe by id, never by payload — identical payloads can be distinct legitimate events.
- **LWW (last-write-wins)** — conflict resolution that keeps the highest-timestamped value (with a deterministic tie-break). Converges — but *discards* the losing concurrent write (our ts=5 "balance=200" silently vanished), and inherits clock quality.
- **CRDT / G-Counter** — Conflict-free Replicated Data Types: state that merges commutatively/associatively/idempotently (G-Counter: per-node counts, merge = componentwise max, value = sum), so AP replicas converge **without losing concurrent updates** — the principled upgrade from LWW.
- **FLP impossibility** — Fischer–Lynch–Paterson: in a fully asynchronous system with even one possibly-crashing process, no deterministic consensus protocol can guarantee termination. Practical protocols assume partial synchrony (timeouts) and sacrifice liveness, never safety.
- **split-brain / divergence** — partitioned sides each accepting writes and developing conflicting state (our AP register reading 200 on one side, 300 on the other). The condition reconciliation (LWW, merges, quorums) exists to repair.

## Step 20 — Spring Events + Kafka, the Outbox Pattern & Real-Time Push (SSE)

- **domain event** — an immutable record of a fact that happened ("a transfer completed"), carrying everything a consumer needs to react without calling the producer back. In Java: a `record`; in Spring: any object handed to `ApplicationEventPublisher` (no base class needed since 4.2).
- **`ApplicationEventPublisher`** — Spring's in-process event bus interface; the `ApplicationContext` itself implements it. Publishing decouples the producer from its listeners — events are routed by the listener method's parameter *type*.
- **`@TransactionalEventListener`** — a listener bound to a transaction **phase** (`BEFORE_COMMIT`, `AFTER_COMMIT` (default), `AFTER_ROLLBACK`, `AFTER_COMPLETION`). Under the hood the event is parked in a `TransactionSynchronization` and delivered when the transaction manager reaches that phase; on rollback, parked AFTER_COMMIT deliveries are discarded.
- **dual-write problem** — needing to update two systems (DB + broker) that share no transaction: a crash between the writes loses an event or invents one. The reason "save then publish" is a bug, not a design.
- **(transactional) Outbox pattern** — write the event as a row in the *same DB transaction* as the business change (atomic), then have a **relay** publish unpublished rows to the broker and mark them only after the broker acks. Converts the dual-write into one write + at-least-once delivery.
- **relay (message relay)** — the Outbox's second half: drains unpublished rows oldest-first, publishes each, marks it published. Ours polls on a `@Scheduled` 2-second timer; the production-grade upgrade is CDC (Debezium tailing the WAL, with an outbox event router).
- **partial index** — an index with a `WHERE` predicate (`… where published = false`): only matching rows are indexed, so the relay's "find unpublished" scan stays tiny while published history grows unbounded.
- **Kafka topic / partition / offset / consumer group** — a topic is a durable, append-only **log** (not a queue — reads don't delete); a partition is a shard of it (ordering only *within* a partition; the record **key** hashes to one); an offset is a consumer group's per-partition bookmark; a consumer group shares partitions among members (scale-out) while *different* group-ids each receive everything (pub-sub).
- **bootstrap servers** — the broker address(es) a Kafka client first contacts to discover the cluster (`spring.kafka.bootstrap-servers`; under Testcontainers, the container's random mapped port via `@ServiceConnection`).
- **`KafkaTemplate`** — Spring's producer façade (`send(topic, key, value)` → `CompletableFuture<SendResult>`); created by Boot's `KafkaAutoConfiguration` — which in Boot 4 lives in the **`spring-boot-starter-kafka`** starter (bare `spring-kafka` gives no bean).
- **`@KafkaListener`** — declares a method as a message handler; Boot's autoconfigured listener container polls the topic on a background thread and invokes the method per record, committing offsets after successful handling (default ack-mode BATCH).
- **`auto-offset-reset: earliest`** — what a consumer group with *no committed offset* does: start from the topic's beginning (vs `latest`, only-new). Why a fresh notification service (or a renamed group-id) replays history.
- **at-least-once delivery** — the broker/relay contract: never lost, possibly duplicated (relay crash between send-ack and mark-published; consumer crash between processing and offset commit). The reason consumers must be idempotent.
- **idempotent consumer** — dedupes by a stable event id before applying effects: `ConcurrentHashMap.newKeySet().add(id)` returning `false` → skip. In-memory for teaching (resets on restart); durable in production (Redis/DB unique constraint — Step 21).
- **exactly-once effect** — the achievable substitute for impossible exactly-once delivery: at-least-once delivery + idempotent processing ⇒ duplicates arrive but don't act. Proven here: received 3, applied 2.
- **poison message** — a record the consumer can never process (e.g. unparseable payload). Rethrowing makes the container redeliver it in a hot loop; the honest fix is a **Dead-Letter Topic** (Step 21) — ours logs-and-skips with a comment admitting it.
- **Server-Sent Events (SSE)** — one-way server→client push over a long-lived HTTP response (`Content-Type: text/event-stream`; frames of `event:`/`data:` lines ended by a blank line). The browser's `EventSource` auto-reconnects. Simpler than WebSocket; perfect for notifications.
- **`SseEmitter`** — Spring MVC's handle on one open SSE response: returning it switches the request to **servlet async** (response parked, request thread freed); any thread may later `send(...)` frames; lifecycle hooks (`onCompletion`/`onTimeout`/`onError`) clean up dead clients.
- **`@ConditionalOnProperty`** — Boot's conditional-bean mechanism (the same machinery autoconfiguration uses), here gating the relay's scheduler so tests switch polling off (`matchIfMissing = true` ⇒ on by default).
- **`fixedDelay` vs `fixedRate`** — `@Scheduled` semantics: fixedDelay waits N ms *after a run completes* (no overlap); fixedRate fires every N ms regardless (runs can pile up). A draining poller wants fixedDelay.
- **Redpanda** — a Kafka-API-compatible broker in a single binary (no ZooKeeper/JVM); pinned `redpandadata/redpanda:v24.2.7` via Testcontainers `RedpandaContainer` + `@ServiceConnection` for real-broker tests.
- **`@ServiceConnection`** — Boot 3.1+ Testcontainers glue: annotate a container bean and Boot derives the connection properties (JDBC URL, `spring.kafka.bootstrap-servers`) automatically — no `@DynamicPropertySource` boilerplate.
- **Awaitility** — the async-assertion library (`await().atMost(…).untilAsserted(…)`): polls an assertion until it holds or times out — the disciplined replacement for `Thread.sleep` in tests of eventually-true behavior.
- **Jackson 2 vs Jackson 3** — `com.fasterxml.jackson.*` vs `tools.jackson.*` (renamed package; the maintained line). Boot 4's web stack defaults to Jackson 3, so a fresh service injects the Jackson-3 `ObjectMapper`, while demand-account constructs its own Jackson-2 mapper — same JSON on the wire, per-module library choice.
- **CDC (Change-Data-Capture)** — turning a database's commit log (Postgres WAL) into an event stream (Debezium); replaces the polling relay with lower latency and no DB-transaction-held sends — the Step-54 upgrade path for the Outbox.

## Step 21 — Payments: the Saga Pattern, Redis Idempotency Keys & Dead-Letter Topics

- **Saga** — a multi-step operation expressed as a sequence of **local transactions** that each commit independently, with **compensating transactions** to semantically undo committed steps when a later one fails. The standard replacement for distributed ACID when no single transaction spans the steps.
- **compensating transaction** — a new, opposite action that undoes a *committed* step's business effect (a refund undoes a debit). Not a rollback: the original work stays in history (our ledger keeps both rows, netting zero). Must itself be durable/retryable — compensation can fail too.
- **orchestration (Saga)** — a central coordinator (`PaymentService`) calls each step and decides compensations on failure. Explicit, testable flow; one log stream; risk of a god-class.
- **choreography (Saga)** — no coordinator: each service reacts to events and emits the next (`debited` → `credited`/`credit-failed`). Maximum decoupling; implicit flow you must trace across services.
- **Saga isolation (lack of)** — between steps, intermediate state is visible to all readers (money left A, not yet at B). A Saga is eventually consistent, never atomic — design with pending states + idempotency.
- **pivot/recovery framing** — steps before the failure are compensated *backward* (our refund); alternatively a Saga can retry *forward* past a pivot step. Ours is a backward-recovery Saga.
- **`Propagation.REQUIRES_NEW`** — transaction propagation that suspends any current transaction and opens a brand-new one that commits when the method returns. Gives each Saga leg its own irrevocable commit. Only applies through the Spring proxy → steps live in their own bean (self-invocation pitfall, Step 7).
- **orchestrator not `@Transactional`** — the deliberate absence: an outer transaction around `REQUIRES_NEW` steps would be an idle, misleading umbrella (nothing durable in it, connection pinned, implies rollback exists). No umbrella transaction *is* the Saga model.
- **Idempotency-Key** — a client-supplied header naming the *attempt*, so a retry of the same attempt returns the original result instead of re-executing. De-facto standard header name (Stripe; IETF draft). Server-generated keys are useless — only the client knows two requests are the same attempt.
- **`SET key value NX EX ttl`** — the Redis idiom behind `setIfAbsent(key, value, ttl)`: store only if the key does **N**ot e**X**ist, with an **EX**piry — one atomic server-side command (Redis executes commands single-threaded, so concurrent claims can't both win). First-write-wins by construction.
- **TTL (idempotency)** — the key's lifetime, sized to the client's maximum retry window (ours: 24 h). Redis deletes expired keys itself — no cleanup job (the operational win over Step 14's DB table).
- **`StringRedisTemplate`** — Boot's auto-configured String-key/String-value Redis client façade; `opsForValue()` exposes the plain GET/SET command group. The generic `RedisTemplate` would JDK-serialize values to binary.
- **Lettuce** — the default Redis driver under `spring-boot-starter-data-redis`: netty-based, thread-safe, one multiplexed connection; connects lazily (first command, not startup).
- **`spring.data.redis.*`** — the Boot 3+ property prefix for Redis connection settings (the old `spring.redis.*` is silently ignored — a classic upgrade trap).
- **`@ServiceConnection(name = "redis")`** — Testcontainers→Boot glue for a `GenericContainer`: since a generic container carries no type info, Boot matches the connection-details factory by *name*; `"redis"` aims `spring.data.redis.*` at the container's random mapped port. Omitting `name` = app silently targets `localhost:6379`.
- **`GenericContainer`** — Testcontainers' any-image container (Redis has no dedicated class in the core modules); `.withExposedPorts(6379)` maps the container port to a random free host port.
- **poison message** — a record the consumer can *never* process (e.g. un-parseable payload). Without a DLT it either blocks the partition (rethrow-forever) or vanishes (swallow-and-log). Step 20's consumer admitted this debt in a comment; Step 21 pays it.
- **Dead-Letter Topic (DLT/DLQ)** — the quarantine topic (`<topic>.DLT`) where the error handler republishes a record after retries are exhausted — value preserved byte-for-byte plus diagnostic headers (original topic/partition/offset, exception message) for forensics and re-drive.
- **`DefaultErrorHandler`** — Spring Kafka's `CommonErrorHandler`: on a listener throw it *seeks the consumer back* to the failing offset (retries are real re-deliveries), counts attempts per record, and on exhaustion invokes its recoverer. Declaring the bean is the whole installation — Boot wires it into the listener container factory.
- **`DeadLetterPublishingRecoverer`** — the "retries exhausted" recoverer: publishes the failed record via a `KafkaTemplate` to a destination chosen by a resolver `BiFunction<record, exception, TopicPartition>` — ours: same topic + `".DLT"`, same partition.
- **`FixedBackOff(0L, 2L)`** — retry policy: zero delay between attempts, max 2 retries ⇒ **3 total attempts**. Right for deterministic failures (a parse error won't heal); transient failures want delay/`ExponentialBackOff`; can't-stall-the-partition cases want `@RetryableTopic` (non-blocking retries).
- **2PC / XA** — two-phase commit: a coordinator collects prepare-votes from all participants, then commits everywhere — atomic but lock-holding, blocking on coordinator failure (*in-doubt* participants), and unsupported by Kafka/most services. The "then" to the Saga's "now."
- **cached test context (TestContext framework)** — Spring caches application contexts keyed by the test-class configuration fingerprint; same-config `@SpringBootTest`s share one context — same singleton beans, counters, and Testcontainers. Feature (speed), and trap: assert **deltas**, not running totals (our `TransferEventConsumerKafkaTest` fix).
- **delta assertion** — measuring `value − before` snapshots instead of absolute state, making a test immune to traffic from other tests sharing a context/fixture — the cheap alternative to `@DirtiesContext`.
- **reserve-then-complete** — the hardened idempotency protocol: atomically claim the key (`SET key PENDING NX EX`) *before* executing; losers of the claim get 409/poll; the winner overwrites `PENDING` with the result; a crashed winner's claim expires via TTL. Closes the concurrent-duplicate window the check-execute-record flow leaves open.

## Step 22 — Caching, Async & Clustered Scheduling (the Market-Info Read Model)

- **cache-aside** — the caching pattern where the *application* checks the cache, loads from the source on a miss, and stores the result. Spring's `@Cacheable` is cache-aside automated by a proxy. Contrast read-through/write-through, where the cache layer itself does the loading.
- **`@Cacheable`** — proxy-applied annotation: on a hit, return the cached value *without running the method*; on a miss, run it and store the result. Inert without `@EnableCaching`, and bypassed by self-invocation (Step 7).
- **`@CachePut`** — always run the method and overwrite the cache entry with its result (TTL restarts). The refresh operation of a read model.
- **`@CacheEvict`** — remove an entry (or a whole cache). The third sibling — invalidation on demand.
- **cache name vs cache key** — the *name* (`fxRates`) is the namespace/configuration unit; the *key* (`USD/EUR`, from a SpEL expression over the method arguments) identifies the entry. In Redis: `fxRates::USD/EUR`.
- **SpEL cache key** — `key = "#base + '/' + #quote"`: evaluated against the arguments **as passed**, *before* the method body runs — so normalization (upper-casing) must happen before the annotated boundary or the cache fragments.
- **`RedisCacheManager`** — Boot's auto-configured `CacheManager` when `spring.cache.type=redis`: each entry is a Redis string key with a TTL; values JDK-serialized by default (hence cache values `implement Serializable`), JSON via a customizer.
- **TTL (time-to-live)** — how long a cache entry exists after being written (`spring.cache.redis.time-to-live: 60s` — unit required; a bare `60` is *milliseconds*). The read model's staleness bound.
- **write-visibility (networked cache)** — a freshly written entry becomes readable only after a network round-trip; read-after-write is not instant. Tests `await` a cache-served read instead of asserting immediately.
- **eventual consistency (of a cache)** — reads may be stale, boundedly: TTL + refresh cadence + write-visibility. You *choose* the tolerance (indicative FX rates: fine; balances: not).
- **CQRS read model** — a read-optimized view separate from the write side. A TTL'd cache is the simplest one; event-sourced projections (Step 52) are the full version.
- **cache stampede (thundering herd)** — a hot entry expires and all concurrent readers miss at once, hammering the upstream — check-then-act as a distributed race. Mitigate: pre-warming/scheduled refresh, jittered early refresh, single-flight loading (`@Cacheable(sync=true)` is per-JVM only).
- **pre-warming** — populating cache entries before traffic needs them (our `@Scheduled` refresh `@CachePut`s the tracked pairs — its first `fixedRate` tick fires immediately at startup, so even the "first" request is a hit).
- **`@Async`** — run the method on a task executor instead of the caller's thread; return `void` or a `Future`/`CompletableFuture` (prefer the future — exceptions land where callers can see them). Needs `@EnableAsync`; silently synchronous without it.
- **`SimpleAsyncTaskExecutor` (+ virtual threads)** — an executor that creates a **new thread per task** (no pool, no queue); with `setVirtualThreads(true)`, each task gets a fresh virtual thread. The historical anti-pattern that became *the* pattern: you don't pool virtual threads.
- **`applicationTaskExecutor`** — the bean name Boot uses for the default application executor; declare your own bean under this name and the auto-configured one backs off, putting `@Async` (and MVC async) on your executor.
- **`Thread.currentThread().isVirtual()`** — JDK 21+ introspection; returning it from an async body turns "we run on virtual threads" from a config claim into a testable boolean.
- **ShedLock** — a small library that makes `@Scheduled` cluster-safe: before a `@SchedulerLock` method runs, the instance must acquire a named lock in a shared store; only the winner runs that tick, others skip (no queue, no retry). A duplicate-*effort* guard, not a correctness mutex (expiry-based, no fencing) — money stays on DB row locks (Step 12).
- **`@SchedulerLock(name, lockAtMostFor, lockAtLeastFor)`** — the per-job guard. `name`: the lock's identity all instances compete for. `lockAtMostFor`: crash bound — auto-release if the holder dies; must exceed worst-case runtime (too short ⇒ overlapping runs). `lockAtLeastFor`: minimum hold — guards against clock-skew/instant-finish double-runs.
- **`LockProvider` / `RedisLockProvider`** — ShedLock's store SPI and its Redis implementation: acquire = atomic `SET key NX` + TTL (the same Redis primitive as Step 21's Idempotency-Key), serialized by Redis's single-threaded execution. `lock()` returns `Optional<SimpleLock>` — present = you own it; empty = skip, non-blocking.
- **`@EnableSchedulerLock(defaultLockAtMostFor = …)`** — registers ShedLock's interceptor; the default is the fallback crash bound for any job that doesn't set its own.
- **`fixedRate` vs `fixedDelay`** — `fixedRate`: start-to-start interval; `fixedDelay`: end-to-start. With no `initialDelay`, the **first tick fires immediately at startup**. The `…String` attribute variants accept `${property}` placeholders.
- **`@ConditionalOnProperty` gate** — registering a whole `@Configuration` only when a property matches (`matchIfMissing = true` ⇒ absent = on). Our idiom for "scheduler on in production, off in tests" — tests then drive the job/lock directly, deterministically.
- **test `application.properties` (merge, don't shadow)** — a test-resources file with a *different name* than the main config merges over it per-key; a same-named file replaces it wholesale (the Step-20 trap).
- **standalone MockMvc** — `MockMvcBuilders.standaloneSetup(controller)`: a minimal DispatcherServlet harness around one hand-constructed controller, **no Spring context** — the lightest web-test rung (below `@WebMvcTest` and full `@SpringBootTest`), ideal when the app's context drags in infrastructure (caching/Redis) a thin controller doesn't need.
- **`@ServiceConnection(name = "redis")`** — on a `GenericContainer`, the `name` declares which service the container provides so Boot can pick the right connection-details factory; omit it and properties silently keep pointing at `localhost` (the Step-21 trap, re-confirmed here).

## Step 23 — Onboarding Orchestration (Coordinating Services with Compensation)

- **orchestration** — coordinating a multi-service workflow with a central **coordinator** that calls each step in order and decides what happens on failure; the flow is explicit in one place (easy to read/test/trace), at the cost of the coordinator knowing every participant.
- **choreography** — the coordinator-free alternative: each service reacts to events and emits the next (Steps 20/21); maximally decoupled, but the flow is implicit across handlers. Orchestrate short critical flows; choreograph broad loosely-coupled reactions.
- **compensating action (compensation)** — a *new forward action* that semantically undoes an already-committed step (deactivate undoes create; refund undoes debit). Required because each step committed in its own service/DB — there is no cross-service rollback. Must be designed, coded, and tested first-class; ideally idempotent (safe to repeat).
- **pivotal vs retriable step** — workflow-step classification: a *pivotal* step's failure triggers compensation of earlier steps; a *retriable* step (e.g. a welcome notification via the outbox) is fire-and-recover and deserves no compensation. Not every step earns an undo.
- **declarative HTTP client (`@HttpExchange`)** — a Java interface whose annotated methods (`@PostExchange("/api/…")`) declare remote calls; Spring generates the implementation as a **JDK dynamic proxy** over a `RestClient` (visible in stack traces as `jdk.proxy2.$ProxyNN`). Parameter annotations are the controller vocabulary in reverse: `@RequestBody`/`@PathVariable`/`@RequestHeader` describe the request being *built*.
- **`HttpServiceProxyFactory`** — the machinery that parses an `@HttpExchange` interface once (path templates, parameter binders, return types) and manufactures the proxy: `HttpServiceProxyFactory.builderFor(RestClientAdapter.create(restClient)).build().createClient(type)`.
- **`RestClient` error default** — the modern blocking client **throws on 4xx/5xx** (`HttpClientErrorException`/`HttpServerErrorException`) unless you replace the status handler — the property an orchestrator's catch-and-compensate relies on; timeouts/connection failures surface as `ResourceAccessException`.
- **connect timeout vs read timeout** — two clocks on every remote call: connect bounds the TCP handshake ("is the host there?"), read bounds waiting for response bytes once connected ("is it thinking forever?"). Set at different layers (JDK `HttpClient` builder vs `JdkClientHttpRequestFactory.setReadTimeout`); a coordinator without them donates a thread to every hung dependency.
- **tolerant reader (`@JsonIgnoreProperties(ignoreUnknown = true)`)** — client response records that keep only the fields they need and ignore the rest, so a downstream can add fields without breaking callers (Postel's law). Without it, Jackson fails on the first unknown property.
- **identity propagation / token relay** — forwarding the caller's `Authorization: Bearer …` header verbatim on downstream calls so the resource server authorizes the *original user*; the relay never mints, parses, or stores the token. Works because bearer tokens are self-contained — which also means the relay is trusted with full replay power until expiry.
- **client-credentials token** — the OAuth2 grant for a service acting on its *own* behalf (jobs, consumers — no user to relay); contrast token relay (user-initiated) and token exchange (RFC 8693 — trade a user token for a narrower downstream-scoped one).
- **502 Bad Gateway** — "I, acting as a gateway, got an invalid response from an upstream server" — the precise status for "a service behind me failed" (vs 500 "I broke" / 503 "I'm overloaded"). Mapped via `@ResponseStatus` on the typed `OnboardingFailedException`, thrown only *after* compensation ran.
- **`@ResponseStatus` exception mapping** — the lightweight alternative to `@RestControllerAdvice`: annotate the exception class and MVC's `ResponseStatusExceptionResolver` answers that status (empty body) when it escapes a controller.
- **action endpoint (verb-as-sub-resource)** — `POST /api/customers/{id}/deactivate`: an explicit state-transition endpoint naming the business operation, instead of a generic `PATCH` of fields; `POST` not `DELETE` because nothing is deleted — status changes, the row (and its KYC history) survives.
- **in-process stub** — a test-only `@RestController` `@Import`ed into the booted test app so the same JVM serves the downstreams' endpoints on the test's random port; clients point at `localhost:<port>` and the stub records what arrived (forwarded headers, compensation calls) and can fail on command. Real HTTP, no Docker, deterministic.
- **dirty checking (JPA)** — modify a *managed* entity inside a transaction and Hibernate diffs it against its load-time snapshot at flush, emitting the UPDATE itself — no explicit `save()` call needed (CIF's `deactivate`).
- **durable workflow** — orchestration whose position is persisted (saga state written before each step) so a crashed coordinator can resume or compensate on restart — what workflow engines (Temporal, Camunda) and event-sourced sagas provide; the documented gap of a synchronous in-memory orchestrator (ADR-0014).

## Step 24 — Spring Batch (EOD Jobs) & the Phase-D Capstone

- **batch processing (vs online)** — high-volume, no-user-waiting work over the whole dataset (the EOD interest run), optimized for throughput and *recoverability* rather than latency; the design question shifts from "respond in 200ms" to "survive a bad record and resume after a crash."
- **EOD (end-of-day) job** — the nightly batch a bank runs over the whole book: interest accrual, statements, reconciliation. Must not abort on one bad record and must not double-apply on re-run.
- **Job / Step (Spring Batch)** — a `Job` is the unit you launch; it's composed of `Step`s. Batch 6 packages: `…batch.core.job.Job`, `…batch.core.step.Step`, builders under `…core.job.builder` / `…core.step.builder`.
- **chunk-oriented step** — the read → process → accumulate-N → write loop; each chunk commits in its own transaction. Chunk size trades throughput (fewer commits) against memory, lock-hold time, and redo-on-rollback.
- **chunk transaction** — the transaction Batch opens around one chunk: the writer's effects (credits, ledger inserts) *and* Batch's own counters/ExecutionContext commit atomically together — which is what makes the restart bookmark trustworthy. The writer is deliberately **not** `@Transactional`; it participates.
- **`ItemReader` / `ItemProcessor` / `ItemWriter`** — the chunk trio (Batch 6: `org.springframework.batch.infrastructure.item.*`). Reader yields items one at a time; processor transforms/filters/rejects; writer receives the whole `Chunk<T>` at once.
- **`RepositoryItemReader`** — Batch's adapter over a Spring Data repository: pages via the named method (`findAll` + a `PageRequest`), requires a **deterministic sort** (`id ASC`), and checkpoints its page/index position under its `name` in the ExecutionContext — a deliberately *stateful* bean.
- **filter (processor returns `null`)** — the cheap "no work needed" path: item dropped, `FILTER_COUNT++`, no exception, no rollback. Model expected no-ops (zero-balance accounts) as filters.
- **skip / `skipLimit`** — fault tolerance for bad records: a configured (allowlisted) exception type is tolerated up to a bounded count, recorded as `*_SKIP_COUNT`. Beyond the limit — or for any unregistered exception — the job fails (mass skips mean the feed is broken).
- **scan (skip isolation)** — what a mid-chunk skip actually costs: the chunk's transaction rolls back, then Batch re-processes the chunk's items **one at a time in mini-transactions** to isolate the offender; survivors commit, `ROLLBACK_COUNT` rises. Why skip is expensive and filter is free.
- **retry / `retryLimit`** — re-attempt an item/chunk on a configured *transient* exception (a lock conflict with live traffic — `OptimisticLockingFailureException`) before skip/fail logic applies. Transient first, tolerable second, fatal last.
- **JobRepository** — the JDBC-backed memory of Spring Batch: every instance, execution, and step execution persisted in the `BATCH_*` tables. Auto-configured by Boot (don't add `@EnableBatchProcessing` — it turns Boot's autoconfig *off*).
- **`BATCH_*` schema (Flyway-owned)** — the six metadata tables + three sequences, created by copying the canonical `schema-postgresql.sql` from the resolved `spring-batch-core` jar into a Flyway migration (V4) with `spring.batch.jdbc.initialize-schema=never`: one schema owner, reviewable history, no startup `CREATE TABLE`s.
- **JobInstance** — the *logical* run: job name + a hash (`JOB_KEY`) of the **identifying** JobParameters; "the interest run for 2026-06-10" is one instance regardless of attempts (unique constraint `(JOB_NAME, JOB_KEY)`).
- **JobExecution** — one *attempt* at an instance (status, timestamps, exit code/message). Restart = a new execution on the same instance.
- **StepExecution** — the per-step scoreboard: `READ_COUNT`, `WRITE_COUNT`, `FILTER_COUNT`, `PROCESS_SKIP_COUNT`, `ROLLBACK_COUNT`, `COMMIT_COUNT` — the numbers an honest batch test asserts alongside the money effects.
- **ExecutionContext** — the serialized state (e.g. the reader's "page 80, index 3") written to the `_CONTEXT` tables at each chunk commit — **the restart bookmark**.
- **identifying JobParameters** — the parameters that define a JobInstance's identity (flagged `IDENTIFYING='Y'`). A unique `runId` per test run → always a fresh instance; a `runDate` in production → "tonight" is one instance forever.
- **restartability** — relaunching a FAILED instance resumes from the ExecutionContext instead of redoing (or double-applying) finished work; requires a deterministic read order. A COMPLETED instance refuses to re-run: `JobInstanceAlreadyCompleteException` — a feature, the guard against double-crediting.
- **`spring.batch.job.enabled=false`** — disables Boot's `JobLauncherApplicationRunner` (which otherwise runs every `Job` bean at startup). A *service containing a job* must boot without side effects; jobs are launched explicitly (`JobLauncher` + parameters) or by an EOD `@Scheduled`+ShedLock trigger.
- **partitioned step** — the scale-out shape: split the keyspace (account-id ranges) across worker StepExecutions running the same reader/processor/writer, each with its own counts and bookmark; a manager step aggregates.
- **exactly-once effect** — the capstone's theorem made real: exactly-once *delivery* is impossible, so engineer at-least-once delivery (transactional Outbox; relay marks published only after the broker ack) + an idempotent consumer (dedupe by eventId). One run shows deliveries ≥ 2 and applications = 1 — delivery count unbounded, effect count pinned.

## Step 25 — SOLID & Clean Code (Refactoring a Smelly Service)

- **SOLID** — five principles for changeable OO design: SRP, OCP, LSP, ISP, DIP. Not a checklist to apply everywhere — a vocabulary for naming why code is hard to change.
- **Single Responsibility Principle (SRP)** — a class has **one reason to change** (one axis of change), not "one method." Our consumer changed when the wire format, the dedup store, the wording, or the Kafka topology changed — four reasons, so it was split four ways.
- **Open/Closed Principle (OCP)** — open to extension, closed to modification: new behaviour (a Redis dedup adapter) arrives as a new class, with zero edits to existing callers. The mechanical consequence of depending on abstractions.
- **Liskov Substitution Principle (LSP)** — any implementation must honour the abstraction's **documented contract**, not just its signature. `markIfNew`'s "atomically" clause is the contract; a `HashSet`-backed adapter would type-check and still violate it under concurrency.
- **Interface Segregation Principle (ISP)** — small, role-specific interfaces beat fat ones; clients shouldn't depend on methods they don't use. Our port has exactly one method because its one consumer needs exactly one operation.
- **Dependency Inversion Principle (DIP)** — high-level policy must not depend on low-level detail; both depend on an abstraction **owned by the policy side**. Before: `Consumer → ConcurrentHashMap`. After: `Consumer → ProcessedEventStore ← InMemoryProcessedEventStore` — both arrows point at the interface.
- **code smell** — a surface symptom that *suggests* (not proves) a design problem: worth a census before any surgery. Smells are hints; the refactoring catalog holds the fixes.
- **god method / god class** — one unit doing many jobs, accreted one sensible addition at a time; the most common smell. Fix: extract collaborators along axes of change.
- **feature envy** — a method more interested in another structure's data than its own (`buildMessage(JsonNode)` reaching into the JSON tree). Fix: move the behaviour to the data's owner (`Notification.from(TransferEvent)`).
- **Law of Demeter** — "talk to your friends, not strangers": avoid chained reaches through foreign structures (`node.get("from").asText()` chains). A locality-of-knowledge heuristic, not an iron law.
- **primitive obsession** — using raw primitives/generic containers (`JsonNode`, loose `String`s) where a named domain type belongs. Fix: introduce the type — our `TransferEvent` record.
- **refactoring** — changing **structure without changing behaviour**, in small steps under a green test suite. Distinct from a rewrite, which may change behaviour. The machine-checkable signature of a true refactor: the tests have a zero-line diff and still pass.
- **behaviour preservation** — the refactor's invariant: every observable output (HTTP responses, log lines, SSE payloads, counters, *and failure modes* like throw-on-poison) is identical before and after. Includes preserving known limitations (our restart-amnesia dedup).
- **safety net (test suite as)** — the pre-existing tests, deliberately left unedited, that judge a refactor. Tests that assert phenomena (counters, broker records, responses) survive structural change; tests coupled to structure shatter — itself a test smell.
- **characterization test** — a test written *before* refactoring legacy code to pin down what it currently does (bugs included), creating a net where none exists. (Feathers, *Working Effectively with Legacy Code*.)
- **seam** — a place where you can substitute behaviour for testing without editing production code. A port is a designed-in seam: any `ProcessedEventStore` — including a lambda — can stand in.
- **port** — an interface **owned by the code that uses it**, named for the caller's need (`markIfNew`), free of technology words. DIP's abstraction, and Step 26's organizing principle.
- **adapter** — a technology-specific implementation of a port (`InMemoryProcessedEventStore`; a Redis one later). Adapters are swappable; the policy never names them.
- **static factory method** — a named static creator (`Notification.from(event)`) used when construction derives values (the message) or deserves a domain name. The `List.of`/`Optional.of` convention.
- **injection by type** — Spring resolves constructor parameters by matching bean types; one bean implementing the port → injected automatically; two → `NoUniqueBeanDefinitionException` at startup (disambiguate via `@Primary`/`@Qualifier`/conditions).
- **YAGNI ("you aren't gonna need it")** — don't build for hypothetical futures; introduce an abstraction when a concrete reason exists (second implementation, test seam, module boundary) — and record the reason (ADR) when you judge the bar met.
- **DRY / KISS** — don't repeat knowledge (the message wording now lives in exactly one place); keep solutions as simple as the problem allows (no `NotificationFactory` class for one static method).

## Step 26 — Hexagonal Architecture (Ports & Adapters) + DDD

- **hexagonal architecture / ports-and-adapters** — Alistair Cockburn's design: a framework-free core (domain + application) surrounded by **ports** (interfaces) into which **adapters** (concrete tech) plug at the edges. The hexagon shape carries no meaning — it's just "more than four sides" to suggest many pluggable edges. Same idea as Clean and Onion architecture, different vocabulary.
- **dependency rule** — the one law that makes it work: **source dependencies point inward**. Adapters depend on application ports; the application depends on the domain; the domain depends on nothing outward. Verifiable, not aspirational — our domain's whole import surface is `import java.math.BigDecimal;` and `javap | grep -c adapter` on the use case is 0.
- **inbound (driving) port** — what the application **offers** the outside world: a use-case interface implemented *by the application* and called *by driving adapters*. `NotifyOnTransfer.handle(TransferEvent)` — named for the use case (a verb phrase from the business), not the mechanism.
- **outbound (driven) port** — what the application **needs** from the outside world: an interface declared *by the application* and implemented *by driven adapters*. `ProcessedEventStore`, `NotificationPublisher` — named for the need ("publish notifications"), never the tech ("SseSender"). Same word "port" as inbound, **opposite** direction of implementation.
- **driving (primary) adapter** — an adapter that *calls* an inbound port, translating an external trigger into a domain call: our `TransferEventConsumer` (Kafka → `NotifyOnTransfer`), a future REST or CLI entry point. It drives the application.
- **driven (secondary) adapter** — an adapter that *implements* an outbound port, satisfying a need the core declares: `InMemoryProcessedEventStore` (implements `ProcessedEventStore`), `SseHub` (implements `NotificationPublisher`). The application drives it through the port.
- **domain (the core ring)** — the pure centre: business model and rules with **no framework, transport, or persistence imports**. Our `TransferEvent` and `Notification` records. Purity means rules survive framework churn, test in microseconds without a container, and never depend outward.
- **application service / use case** — the policy layer that implements inbound ports and orchestrates the domain *through outbound ports only* — `NotificationService` (dedupe → build → publish). It depends inward (domain) and sideways (its own ports), never on an adapter. Carries a pragmatic `@Service` stereotype; the domain stays annotation-free.
- **value object** — a DDD tactical pattern: an immutable type defined by its attributes, with equality-by-value, not by identity. Our `TransferEvent`/`Notification` records are textbook value objects (records give immutability + generated `equals`/`hashCode` for free). Two with the same fields *are* the same fact.
- **aggregate (and when to omit)** — a DDD consistency boundary with a root entity guarding invariants. **Deliberately absent** from the notification context: a thin read/push model with no invariant to guard doesn't earn one. The richer aggregates live in the money domain (demand-account's `Account` with `FOR UPDATE` locks). DDD patterns apply where the domain warrants them, not ceremonially.
- **bounded context** — the scope within which a domain model and its language are valid. Notification is its own thin context; demand-account is another (richer) one — the same word "transfer" carries a different model in each.
- **DDD tactical patterns** — the implementation-level toolkit: value objects, entities, aggregates, repositories, domain services, application services. *Tactical* (the building blocks) vs *strategic* (bounded contexts, context maps). This step used the tactical slice, right-sized.
- **behaviour-preserving restructure** — moving/renaming structure (here: a multi-package move into rings) while every observable output stays identical. Proven by an **imports-only test diff** that still passes green, plus a mutation check confirming the new layer is on the hot path (we broke the use case's dedup → `expected: 2 but was: 3`, then reverted).

## Step 27 — Enforcing Architecture: ArchUnit + Spring Modulith (Fitness Functions)

- **architecture fitness function** — an automated test that asserts an architectural *characteristic* (here, structural dependency rules) so the design can't erode silently: a violating commit fails the build, not just review. Term from *Building Evolutionary Architectures* (Ford/Parsons/Kua). The shift is from architecture-as-wiki-diagram (drifts) to architecture-as-executable-test (verified every run).
- **ArchUnit** — a Java library for writing architecture rules as a fluent DSL over imported **bytecode** (`@AnalyzeClasses`, `@ArchTest static final ArchRule`). Used for *bespoke* rules about a specific design — e.g. our notification hexagon's layer rules. The `archunit-junit5` artifact runs each rule as its own JUnit test (4 rules → `Tests run: 4`).
- **`layeredArchitecture` (ArchUnit DSL)** — a prebuilt rule builder for ring/layer designs: name layers (`.layer("Domain").definedBy("..domain..")`) then state access (`.whereLayer("Domain").mayOnlyBeAccessedByLayers("Application","Adapter")`). We model the hexagon with **Adapter as one layer** so the documented intra-adapter web→push SSE coupling is allowed while any adapter→core-inward violation still fails. `.consideringAllDependencies()` makes it check every reference kind.
- **bytecode analysis** — analysing compiled `.class` files (not source). Key consequence: an **unused `import` is erased by the compiler**, so it's invisible to the analysis — only a *real* reference (annotation, field type, parameter, method call, supertype) is a dependency. Correct semantics, occasionally surprising; it's why the §12.3 mutation adds a real `@Component`, not just an import.
- **Spring Modulith** — a Spring project that *derives* a module model from package structure and verifies modular-monolith rules (ArchUnit is its engine). `ApplicationModules.of(App.class).verify()` checks **no cycles** + **no internal-package access**; `Documenter` generates living docs. Optional runtime features (module events, observability) exist but we kept it **test scope only** this step.
- **application module (Spring Modulith)** — each **direct sub-package** of the application's base package is one module (demand-account → batch, client, domain, event, outbox, payment, service, web, webhook = 9). By default all of a module's types are **API**; types under an `internal` sub-package are hidden.
- **module cycle** — a dependency loop between modules (A→B and B→A, directly or transitively). The thing that turns a modular monolith into a big ball of mud. `verify()` fails on any cycle, printing the exact offending references on each leg. Our graph is a **DAG**, so it passes.
- **API vs `internal` / named interface** — a module exposes an **API** (its public types) and may hide implementation under an `internal` sub-package; a cross-module reference to another module's `internal` types fails `verify()`. `@NamedInterface` lets you publish a *curated* API and declare `allowedDependencies` against it (the stretch goal). We left demand-account at the default all-API this step.
- **living documentation** — docs *generated from the verified model* so they can't drift from the code. Modulith's `Documenter` writes a C4 component diagram (`components.puml`) + per-module diagrams + per-module canvases into `target/spring-modulith-docs/` on every test run.
- **module canvas** — a per-module `.adoc` document (e.g. `module-service.adoc`) listing the module's base package, its Spring components, its **bean references** (which other modules it touches, by module), and events published/listened-to — the dependency edges spelled out in prose.
- **C4 component diagram** — the level-3 view in the C4 model (Context → Container → Component → Code). Modulith renders the module graph as a C4 component diagram; isolated modules with no inter-module edges (e.g. `client`) are omitted from it but still counted by `verify()`.
- **BOM (Bill of Materials), import scope** — a POM imported via `<type>pom</type><scope>import</scope>` into `dependencyManagement` that curates mutually-compatible versions; children then declare artifacts with **no `<version>`**. We import `spring-modulith-bom` so the Modulith deps need no explicit version (ArchUnit, with no BOM here, is pinned via the `archunit.version` property — note the asymmetry).
