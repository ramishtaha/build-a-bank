## Step 11 — Concurrency & Thread Safety in Java

1. 🌟 **"Is `i++` atomic? Why not?"** *(the classic)* — No. `i++` (and `balance += amount`) is a **read-modify-write**: read the field, add, write it back — three operations, not one. Two threads can both read the same old value, both add, and both write back the same result, so one update is silently lost. Make it atomic with `AtomicInteger.incrementAndGet`, a `synchronized` block / `Lock`, or `LongAdder` for hot counters.

2. 🌟 **"`volatile` vs `synchronized`?"** *(gotcha)* — `volatile` gives **visibility + ordering** (a write is immediately visible to later reads; it's a memory barrier so reads/writes aren't reordered across it) and makes single reads/writes of the field atomic *even for `long`/`double`* — but it does **not** make a compound action atomic, so `volatileCounter++` still loses updates. `synchronized` gives **mutual exclusion + visibility + ordering** across a whole block. Rule: `volatile` for a one-writer flag (e.g. a `stop` signal); `synchronized` when you need atomicity across multiple steps or fields.

3. **"What is happens-before, and what creates it?"** — A guarantee that one action's writes are **visible to** and **ordered before** another action. You never get it for free; specific constructs create the edges: monitor **unlock → lock** of the *same* monitor, **`volatile` write → read** of the same field, **`Thread.start()`** → the started thread's first action, a thread's actions → another thread's successful **`join()`**, and **`final`-field** publication of a correctly constructed object.

4. **"`AtomicLong` vs `synchronized` vs `LongAdder` — when each?"** *(applied)* — `AtomicLong` (lock-free CAS) for a single counter at low/medium contention — no blocking, just occasional retries. `LongAdder` for a *hot* counter at high contention — it stripes the count across cells so threads usually hit different memory (no CAS contention / false sharing), and `sum()` aggregates. `synchronized`/`Lock` when you must update **multiple fields** atomically (an invariant a single CAS can't express).

5. 🌟 **"What are virtual threads, and what do they change?"** *(version-evolution)* — Lightweight threads (stable Java 21, JEP 444) scheduled by the JVM onto a small pool of **carrier** platform threads. When a virtual thread **blocks** (I/O, `sleep`, most `j.u.c` waits) it **unmounts** the carrier, freeing it to run another — so blocking is cheap and you can run millions, writing simple blocking code instead of reactive callbacks. They do **not** change the memory model and do **not** make racy code safe; you no longer pool them (one thread per task).

6. **"How would you prevent two simultaneous withdrawals from overdrawing an account?"** *(concurrency + money — the Step 12 setup)* — Make check-and-debit **atomic**. Inside one JVM: a lock (`synchronized`/`ReentrantLock`) or an atomic compound op. Across instances (the real bank): push the atomicity to the database — a pessimistic row lock (`SELECT … FOR UPDATE`, Step 10), an optimistic `@Version` with retry (Step 9), or `SERIALIZABLE` + retry — or a distributed lock. In-JVM `synchronized` alone only works with a single instance.

> **Behavioral / STAR seed:** *"Tell me about a hard bug you debugged."* — A race that only failed under load. **S/T:** intermittent balance corruption in prod, green on the laptop. **A:** reproduce it *deterministically* (a `CyclicBarrier` that forces the interleaving) so it fails every run, fix with the right primitive, add a stress test. **R:** regression-proof, and a reusable pattern for the team.

## Step 12 — Demand Account, the Double-Entry Ledger & Transactions Deep

1. 🌟 **"Two withdrawals hit the same account at the same instant — what happens, and how do you make it correct?"** *(the money question)* — Without protection it's a lost-update / overdraft race (the balance is a read-check-write). Fix by making check-and-debit **atomic on the contended row**: a pessimistic row lock (`SELECT … FOR UPDATE`) to serialize, optimistic `@Version` + a retry loop, or `SERIALIZABLE` + retry. For a **hot** account, pessimistic; for **rare** conflicts, optimistic.

2. 🌟 **"Optimistic vs pessimistic locking — trade-offs?"** *(the most common)* — Optimistic (`@Version`): no lock during the read, detect the conflict at write time (the `UPDATE … WHERE version = ?` matches 0 rows → `ObjectOptimisticLockingFailureException`) and **retry**; best when conflicts are rare. Pessimistic (`SELECT … FOR UPDATE`): take a row lock at read time, others **block**; best under high contention when you must serialize. Measured here: 20/20 transfers succeed with `FOR UPDATE`; **17/20 fail** with only `@Version`.

3. **"What does `@Transactional(propagation = REQUIRES_NEW)` do, and when would you use it?"** — Suspends the caller's transaction and runs in a **brand-new** one that commits independently — so e.g. an **audit/log** row survives even when the business transaction rolls back. (The default `REQUIRED` joins the caller's transaction and rolls back with it.) It only works across a **bean boundary**: a `this.`-call is self-invocation and bypasses the proxy.

4. **"Does `@Transactional` roll back on a checked exception?"** *(gotcha)* — No. By default it rolls back only on `RuntimeException`/`Error`; **checked exceptions commit** unless you set `rollbackFor`. (That's why `InsufficientFundsException extends RuntimeException` — a failed transfer rolls back automatically.) Bonus gotcha: `@Transactional` is proxy-based, so a self-invocation (`this.method()`) starts no transaction at all.

5. 🌟 **"Why `BigDecimal` for money, not `double`?"** — `double`/`float` are binary floating point and can't represent decimals like `0.10` exactly, so cents drift over many operations. `BigDecimal` is exact decimal: store as `numeric(19,4)`, **compare with `compareTo`** (not `equals`, which is scale-sensitive: `100.00` ≠ `100.0000`), and round explicitly with a `RoundingMode`.

6. **"How do you avoid deadlocks when a transfer locks two rows?"** *(concurrency)* — Acquire the locks in a **consistent global order** (here, by account number) regardless of transfer direction, so A→B and B→A both lock the lower-numbered account first and can't form a hold-and-wait cycle. (Postgres also detects deadlocks and aborts one with `40P01`, but ordering avoids them entirely.)

7. **"Why a double-entry ledger, and how do you know the books balance?"** *(domain + design)* — Every movement is two equal entries — a DEBIT on the payer, a CREDIT on the payee — sharing one `transactionId`, on an **append-only** table. Because every debit has a matching credit, the sum of all entries (credits − debits) is always **zero**; assert `ledgerNet == 0` plus `totalSystemBalance` constant to prove correctness. The materialized `balance` is a fast-read projection kept in sync inside the same transaction (event sourcing in Step 52 makes the ledger the only source of truth).

> **Behavioral / STAR seed:** *"Tell me about a time you prevented a serious bug before production."* — The overdraft race. **S/T:** concurrent transfers could double-spend (a TOCTOU bug), green on a single thread. **A:** reproduced it *deterministically* with a `CyclicBarrier` stress test that lost an update every run, added pessimistic `SELECT … FOR UPDATE` with deadlock-safe lock ordering, and a money-conservation assertion. **R:** the test now fails-closed and the books always balance under 20-thread load.

## Step 13 — Spring MVC / REST Deep (Problem Details, OpenAPI, Filters vs Interceptors)

1. 🌟 **"Walk me through what happens when a request hits a Spring MVC controller."** *(the classic)* — Servlet **filter chain** → **`DispatcherServlet`** (front controller) → **`HandlerMapping`** finds the `@RequestMapping` method → interceptors' **`preHandle`** → **`HandlerAdapter`** binds the arguments (an `HttpMessageConverter` deserializes `@RequestBody`, `@Valid` triggers Bean Validation) and invokes your method → the return value is serialized by a converter (Jackson → JSON) → `postHandle`/`afterCompletion`. Anything thrown goes to the `HandlerExceptionResolver`s — which include your `@ControllerAdvice`.

2. 🌟 **"How do you do global error handling, and what is RFC 9457?"** *(very common)* — A `@RestControllerAdvice` (a `@ControllerAdvice` + `@ResponseBody`) with `@ExceptionHandler` methods that return a **`ProblemDetail`**. Extend **`ResponseEntityExceptionHandler`** so the framework's built-in MVC exceptions (validation, unreadable body, 404/405) also come back as Problem Details. **RFC 9457** (successor to 7807) standardizes the media type **`application/problem+json`** and the body — `type` (problem-kind URI), `title`, `status`, `detail`, `instance`, plus custom extension members — so clients parse errors reliably instead of string-matching bespoke JSON.

3. 🌟 **"Filter vs interceptor — when do you use each?"** *(gotcha)* — A **`Filter`** is at the **servlet-container** level (wraps the whole `DispatcherServlet`), sees **every** request (even ones that 404 with no handler), and has **no handler context** — use it for correlation ids, CORS, compression, request/response wrapping. A **`HandlerInterceptor`** is at the **Spring-MVC** level, runs around the **matched handler**, and **knows which handler** ran (`preHandle`/`postHandle`/`afterCompletion`) — use it for per-handler timing, auth that needs handler metadata, MDC setup. Key detail: **filters auto-register** (just be a bean); **interceptors must be registered** via `WebMvcConfigurer.addInterceptors(...)`.

4. **"Is a `@RestController` thread-safe? What about your filter/interceptor?"** *(concurrency — Step 11 callback)* — Controllers, `@ControllerAdvice`, filters, and interceptors are **singletons** shared across all request threads, so they must be **stateless**: no mutable instance fields holding per-request data (that's a race). Per-request state belongs in **request attributes** (`request.setAttribute(...)`, as our `TimingInterceptor` does), `ThreadLocal`/`MDC`, or method locals — never an instance field.

5. **"How is your OpenAPI/Swagger doc generated, and what changed across versions?"** *(version-evolution)* — **springdoc-openapi** introspects the `RequestMappingHandlerMapping`, request/response types, and validation annotations to emit **OpenAPI 3.1** at `/v3/api-docs` and serve **Swagger UI** at `/swagger-ui.html`; you only supply metadata via an `OpenAPI` bean. The old **springfox** is unmaintained and stuck on older Spring — don't use it. springdoc **3.0.x** supports Spring Boot 4 / Framework 7 (2.8.x targets Boot 3), so it's pinned (it's not in Boot's managed BOM).

6. **"How do you return the right status for a business error like insufficient funds?"** *(applied)* — Map the **domain exception** in the advice to a `ProblemDetail` with the right code (**422 Unprocessable Entity** — well-formed but can't be fulfilled — vs **400** for a malformed request), a stable `type` URI, and a **safe `detail`** (never a raw stack trace, SQL, or secret). Validation failures → 400 with a per-field `errors` map attached via `problem.setProperty("errors", ...)`.

> **Behavioral / STAR seed:** *"Tell me about a time you improved an API's developer experience."* — **S/T:** clients integrated against ad-hoc `{"error":...}` JSON and had to read source to learn the endpoints. **A:** replaced the error envelope with **RFC 9457 Problem Details** (machine-readable `type`/`status` + a per-field `errors` map) and added a generated **Swagger UI** + correlation-id header for support. **R:** client integration bugs dropped, onboarding sped up, and incidents became traceable by request id.

## Step 14 — API Design, Versioning, Idempotency & Webhooks

1. 🌟 **"How do you make a payment API safe to retry?"** *(the fintech classic)* — An **`Idempotency-Key`** header + a server-side store of `key → result`. The client sends a unique key per logical operation on every attempt; the server records the key→result the first time and, on any retry with that key, returns the **stored** result without re-executing. So a timeout-then-retry can't double-charge. Keys get a **TTL** (you don't remember them forever), and the store's **unique constraint** is the concurrency guard for racing duplicates.

2. 🌟 **"How would you secure an outbound webhook?"** *(security)* — Sign each delivery with **HMAC-SHA256** over `timestamp + "." + body` using a **per-partner shared secret**; send the signature + timestamp as headers. The receiver recomputes the HMAC and compares in **constant time** (`MessageDigest.isEqual` — never `equals`, which leaks timing), and **rejects timestamps outside a tolerance window** (replay protection). Deliver over HTTPS and rotate secrets. This is the Stripe/GitHub scheme. HMAC (not a plain `hash(secret∥body)`) avoids length-extension attacks.

3. **"Webhook delivery semantics — and the consequence?"** *(gotcha)* — **At-least-once**: you retry on failure, so a receiver may see the **same event twice** → receivers **must be idempotent** (dedupe by event id). True exactly-once *delivery* is impossible in general; you get exactly-once **effect** via idempotent receivers, plus the **Outbox pattern** (Step 20) to avoid lost/dual writes.

4. **"How do you version an API without breaking clients?"** *(applied)* — Pick a strategy — **URI** (`/api/v1`, visible/cacheable/curl-able — chosen here), **header** (`Accept: …v1+json`, clean URLs but invisible), or **media-type** (most RESTful, heaviest). Add new versions **additively**; when retiring an old endpoint, keep it working and **deprecate gracefully** with `Deprecation: true`, a `Sunset` removal date, and a `Link; rel="successor-version"` (RFC 8594) — never a flag-day removal.

5. **"Offset vs cursor pagination?"** — **Offset** (`page`/`size`) is simple and supports random page access, but degrades at depth (`OFFSET 1_000_000` scans and discards a million rows) and can skip/duplicate rows if data changes between pages. **Cursor/keyset** ("give me 20 after id X / createdAt T") is stable and fast at any depth, at the cost of no random access. Either way, return a **stable envelope you own** (a `PageResponse` DTO), never the ORM's `Page` (its JSON shape is an unstable internal detail Spring warns against exposing).

6. 🌟 **"Two retries of the same idempotent transfer race — what happens?"** *(concurrency)* — Both may **miss** the key lookup and both execute the transfer, but the idempotency key's **PRIMARY KEY** lets only one commit the key row; the other's `INSERT` violates the unique constraint and its **whole transaction rolls back** (undoing its transfer too). Exactly one transfer commits. The database is the coordination point — the same principle as Step 12's `SELECT … FOR UPDATE`.

7. **"Why constant-time signature comparison?"** *(gotcha)* — A normal `equals`/`==` returns as soon as it hits a mismatched byte, so the response time leaks how many leading bytes matched — letting an attacker brute-force a valid signature byte by byte. `MessageDigest.isEqual` examines all bytes regardless of where the first mismatch is, removing the timing signal.

> **Behavioral / STAR seed:** *"Tell me about a time you prevented a costly bug."* — **S/T:** the transfer API's `POST` wasn't idempotent, so a client retry after a timeout could **double-charge**. **A:** added a Stripe-style `Idempotency-Key` + a key store (PK = the concurrency guard), proved with a test that a same-key retry moves money **once**, and HMAC-signed the outbound webhooks (with a replay window) so partners couldn't be spoofed or replayed. **R:** money-moving became safe to retry under both sequential and concurrent duplicates, all test-proven, with the dual-write gap documented for the Outbox follow-up.

## Step 15 — API Gateway / BFF & Service-to-Service HTTP

1. 🌟 **"Draw a microservices architecture — where's the gateway and what does it do?"** *(system design — the opener)* — A single **front door** every external client hits; it **routes** each request to the right internal service and centralizes **edge concerns** (authentication/authorization, rate limiting, TLS termination, correlation-id origination, request/response shaping) so each service doesn't reinvent them. It keeps internal services **private** and decouples clients from the topology (services can split/merge behind it). What does **not** belong in it: **business logic** — keep the gateway a thin edge. A **BFF** is the same idea specialized per client (web/mobile), often aggregating several service calls into one tailored response.

2. 🌟 **"How do services call each other, and what's the modern Spring way?"** — Two styles: **synchronous** over HTTP (request/reply — immediate but *temporally coupled*) or **asynchronous** via messaging (Kafka — decoupled, eventually consistent). For sync in Spring today: a **declarative HTTP interface** — an `@HttpExchange`/`@GetExchange` interface whose implementation is generated by `HttpServiceProxyFactory` over a `RestClient`. Type-safe, no hand-written plumbing, no extra dependency (it's in Spring Framework 6.1+/7) — the successor to `RestTemplate` (maintenance) and OpenFeign (for in-Spring use).

3. 🌟 **"What happens when a downstream service is slow?"** *(the resilience gotcha)* — Without a **timeout**, the caller's thread blocks waiting; a chain of such blocked calls **exhausts the thread/connection pool** and the slowness **cascades** into a system-wide outage. The baseline fix is **connect + read timeouts** so a slow dependency **fails fast** (Spring surfaces it as `ResourceAccessException`) and frees the thread. A connect timeout **alone** isn't enough — a downstream that accepts the connection then stalls still hangs you, so you need the **read** timeout too. The next layers — **circuit breakers, bulkheads, retries with backoff, fallbacks** — are Resilience4j (Step 37).

4. **"Reactive vs servlet Spring Cloud Gateway — which and why?"** *(version/architecture)* — The **reactive** gateway (`-server-webflux`, WebFlux/Netty) scales connections on an event loop — great for very high fan-out/streaming, but a different programming model. The **servlet** gateway (`-server-webmvc`, Spring MVC + a blocking HTTP client) fits an MVC stack and, with **virtual threads**, scales blocking forwards without the reactive complexity. We use **servlet** because the platform is MVC + virtual threads. (Config prefix: `spring.cloud.gateway.server.webmvc.routes` since Spring Cloud 2025; the old `spring.cloud.gateway.mvc.routes` is deprecated.)

5. **"Sync HTTP vs async messaging — when each?"** — **Sync** when the caller needs an immediate answer and can't proceed without it (and you accept temporal coupling). **Async** when you want decoupling/resilience and can tolerate eventual consistency — events, the **Outbox** pattern, and **Saga** (Steps 20–21). Most real systems use both: sync for north-south request/reply, async for east-west workflows.

6. **"How would you secure the edge?"** *(security)* — Authenticate/authorize **at the gateway** (Steps 16–17), **rate-limit** (Step 51), **terminate TLS**, **strip spoofable inbound headers** (so clients can't forge gateway-added identity/correlation headers), and keep internal services **unreachable from outside** (NetworkPolicy, Step 42). The gateway is the perimeter; also gate `/actuator/gateway/routes` so you don't leak your internal topology.

7. **"What does `StripPrefix` do, and why route by a service prefix?"** *(applied)* — A routing scheme: external `/cif/**` and `/bank/**` carry a **service prefix** that selects the route; `StripPrefix=1` removes that leading segment before forwarding, so the downstream receives its **own** path (`/cif/api/customers/1` → cif's `/api/customers/1`). It gives clients one address per logical area while each service keeps its native API.

> **Behavioral / STAR seed:** *"Tell me about a time you prevented a cascading failure."* — **S/T:** an inter-service HTTP call had **no read timeout**, so a slow dependency could tie up caller threads until exhaustion. **A:** added connect + read timeouts (fail fast → `ResourceAccessException`) on the declarative client, proved it with a test where a 1.5 s stub trips a 400 ms timeout, and flagged circuit breakers/bulkheads as the next layer (Resilience4j, Step 37). **R:** a downstream slowdown stopped propagating into a caller-wide outage, with the resilience roadmap documented.

## Step 16 — Spring Security Deep I (Filter Chain, JWT & Password Encoding)

1. 🌟 **"How does Spring Security work — what's the filter chain?"** *(the classic opener)* — At heart it's **a chain of servlet filters** placed in front of your app (one `FilterChainProxy` delegating to an ordered `SecurityFilterChain`). Every request runs the chain *before* the `DispatcherServlet`/controller. Two jobs, in order: **authentication** (establish identity — here decode+validate the Bearer JWT; none/invalid → **401** via the `AuthenticationEntryPoint`) then **authorization** (apply access rules; not allowed → **403** via the `AccessDeniedHandler`). You configure it by declaring a **`SecurityFilterChain` bean** built from `HttpSecurity` with the lambda DSL — the modern replacement for `WebSecurityConfigurerAdapter`, which was **removed in Spring Security 6**.

2. 🌟 **"401 vs 403 — what's the difference?"** *(the gotcha)* — **401 Unauthorized** = *not authenticated*: no credentials, or invalid/expired ones (the request couldn't establish who you are). **403 Forbidden** = *authenticated but not authorized*: we know who you are, but you lack the required role/permission. They surface at different points of the chain (`AuthenticationEntryPoint` vs `AccessDeniedHandler`). Watch the leakage angle too: a bare 403 can reveal a resource exists — keep responses generic.

3. 🌟 **"What's a JWT and how do you validate it?"** — A **JSON Web Token**: three base64url parts — `header.claims.signature`. The header names the algorithm; the claims carry `sub` (subject), `roles`, `iss`, `iat`, `exp`; the signature (HMAC-SHA256 here) lets a key-holder verify it wasn't forged/tampered. To validate: check the **signature** (with the shared secret or the issuer's public key) **and** the **expiry**, then map claims → authorities. As an **OAuth2 resource server**, Spring's `BearerTokenAuthenticationFilter` + `JwtDecoder` do this per request — **stateless**, no session. A JWT is **signed, not encrypted** — the payload is readable, so never put secrets/PII in claims.

4. **"Symmetric vs asymmetric JWT signing — which and when?"** *(architecture/version)* — **HMAC (HS256)**: one shared secret signs *and* validates. Simple and self-contained for **one issuer + one validator** — but every validator holding the secret can also **forge** tokens. **Asymmetric (RSA/EC)**: the issuer signs with a **private** key; each service validates with the **public** key (fetched from a **JWKS** endpoint, `/.well-known/jwks.json`) and *cannot* mint tokens — least privilege. Move to asymmetric when **many independent services validate** (Step 17/41). We use HMAC now (one auth service) and pin the algorithm on the decoder.

5. 🌟 **"How do you store passwords?"** — Never plaintext, never a fast unsalted hash (MD5/SHA-1 — crackable at scale, rainbow tables). Use a **slow, salted, one-way** hash: **BCrypt** (`$2a$10$<salt><hash>` — cost factor + per-hash random salt baked in), or Argon2/scrypt. Verify with `PasswordEncoder.matches(raw, hash)` (constant-time, never `equals`). The same password hashes **differently** each time (distinct salts) yet both verify — that's what defeats rainbow tables. Tune the cost factor up as hardware improves.

6. **"Do you need CSRF protection for a JWT API?"** — **No** for a **stateless Bearer-token** API: CSRF attacks ride **ambient** credentials (cookies/session the browser sends automatically), and a token API has none — the client must *explicitly* attach the token, so there's nothing to forge (and leaving CSRF on would block non-browser clients). **Yes** for **cookie/session** flows. The rule: *no ambient credentials → no CSRF risk* — and know which mode you're in, because disabling CSRF on a cookie app is a real vulnerability.

7. **"Why is there no auth code inside your `/admin` controller method?"** *(applied)* — Because the **filter chain** already enforced `requestMatchers("/api/auth/admin").hasRole("ADMIN")` *before* the method runs — a non-admin got a 403 from the `AuthorizationFilter` and never reached the controller. Authorization lives in **one** place (the chain, or method security via `@PreAuthorize` — not duplicated ad-hoc in the body), so the rule can't drift out of sync. The controller just *reads* the identity (`Authentication` injected from the `SecurityContextHolder`), it doesn't *establish* it.

> **Behavioral / STAR seed:** *"Tell me about adding security to a system."* — **S/T:** the bank's services were entirely unauthenticated; we needed identity that scales across microservices without server sessions. **A:** stood up a dedicated `services/auth` with a stateless `SecurityFilterChain` (CSRF off — Bearer tokens, no ambient creds), BCrypt-hashed passwords, and short-lived HS256 JWTs issued on login and validated as an OAuth2 resource server; mapped a `roles` claim to authorities for `hasRole` checks. **R:** proved 401 (no token) / 403 (wrong role) / 200 (authorized) over real HTTP plus a §12.3 mutation that breaks the 403 test, and documented the next moves — asymmetric JWKS signing + refresh tokens (ADR-0008, Step 17).

## Step 17 — Spring Security Deep II (Resource Servers, RS256/JWKS, Method Security & Modern Auth)

1. 🌟 **"How do multiple services validate the same JWTs without sharing a secret?"** *(the system-design opener)* — **Asymmetric signing + JWKS.** The issuer (auth) signs with a **private** RSA key (RS256) and publishes the matching **public** key as a **JWKS** (JSON Web Key Set) at a well-known URL (`/oauth2/jwks`). Each service fetches and **caches** the public key (Spring's `jwk-set-uri`) and validates token signatures **locally** — no shared secret to leak, no per-request call back to auth. Tokens carry a `kid` so verifiers pick the right key, which makes **rotation** seamless. This is how OAuth2/OIDC providers (Google, Auth0, Keycloak) work.

2. **"HMAC vs RSA (HS256 vs RS256) for JWT — when each?"** *(gotcha)* — **HMAC (HS256)**: one shared secret signs *and* validates — simplest, but every validator can also **forge**, so it's only safe within a **single** service. **RSA/EC (RS256/ES256, asymmetric)**: the issuer signs with the **private** key; many services validate with the **public** key (via JWKS) and *cannot* mint tokens — least privilege. Switch to asymmetric the moment **more than one independent service** validates the same tokens.

3. 🌟 **"What is an OAuth2 resource server?"** — A service that authorizes requests by validating a **Bearer access token** (signature + expiry + scopes/claims) on **each request, statelessly**, against a configured issuer — no login UI, no session, no cookies. In Spring you set `spring.security.oauth2.resourceserver.jwt.jwk-set-uri` and `http.oauth2ResourceServer(oauth2 -> oauth2.jwt(...))`; Boot wires a `BearerTokenAuthenticationFilter` + a `JwtDecoder` (built from the JWKS) that runs before your controller. demand-account became one this step.

4. **"URL authorization vs method security — when each, and what's the catch?"** — **URL rules** (`authorizeHttpRequests().requestMatchers(...)`) gate the **HTTP edge** coarsely (e.g. "all `/api/**` need a token"). **Method security** (`@EnableMethodSecurity` + `@PreAuthorize("hasRole('ADMIN')")`) expresses **fine-grained, domain-level, reusable** rules right on a method. Real systems use **both**. *The catch:* method security wraps the bean in an **AOP proxy** and applies to **direct in-process calls too** — so annotating a *service* method forces a `SecurityContext` into every unit test that calls it directly. We deliberately kept authz at the edge + a narrow `@PreAuthorize` demo to avoid that blast radius (ADR-0009).

5. **"How do you revoke a JWT?"** *(gotcha)* — Mostly **you can't** before it expires — a self-contained JWT carries no server-side lookup. Mitigations: **short-lived** access tokens (minutes) + a longer-lived, **revocable refresh token** to mint new ones; a **`jti` denylist** (reintroduces some state); and **key rotation** via JWKS (rotating the signing key invalidates everything signed with the old one). If you need *instant* revocation, use **opaque tokens + introspection** (a per-request call to the issuer) instead.

6. **"What's the algorithm-confusion attack, and how does Spring defend?"** *(security gotcha)* — In an RS256 deployment the verification key is **public**. An attacker crafts an **HS256** token using that *public key as the HMAC secret*; a naive verifier that trusts the token's own `alg` header would validate it. Defense: **pin/derive the expected algorithm from the key, never from the token.** Spring's `NimbusJwtDecoder` built from a JWKS validates against the RSA key type, so an HS256 token simply doesn't match — and you never accept `alg: none`.

7. **"What are passkeys, MFA, and step-up auth?"** *(modern)* — **MFA**: a second factor (something you *have*/*are*) beyond the password. **Passkeys/WebAuthn**: phishing-resistant, **device-bound public-key** credentials — the device's biometric unlocks a private key that signs a server challenge; nothing phishable leaves the device. **Step-up**: requiring a **stronger/fresh** factor for a sensitive action (e.g. a large transfer) even though you're already logged in. Spring Security 7 models *how* you authenticated as `FACTOR_*` authorities, so a `@PreAuthorize("hasAuthority('FACTOR_OTP')")` can demand step-up.

> **Behavioral / STAR seed:** *"Tell me about securing a distributed system."* — **S/T:** the bank had one auth service signing HMAC JWTs, but a second service (money movement) now had to trust them — and a shared secret would let any service **forge** tokens. **A:** moved auth to **asymmetric RS256** signing + a published **JWKS**, made demand-account an **OAuth2 resource server** validating via `jwk-set-uri` (public key only), and added URL + `@PreAuthorize` authorization (with a deliberate edge-vs-service blast-radius choice, ADR-0009). **R:** proved a **live cross-service flow** — the money service fetched auth's JWKS, accepted an auth-minted token (201), rejected an unauthenticated request (401), and enforced ADMIN (403/200) — cryptographic trust across services with **no shared secret**, plus a §12.3 mutation showing the auth rule is load-bearing.
