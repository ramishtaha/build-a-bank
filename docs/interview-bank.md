## Step 11 — Concurrency & Thread Safety in Java

1. 🌟 **"Is `i++` atomic? Why not?"** *(the classic)* — No. `i++` (and `balance += amount`) is a **read-modify-write**: read the field, add, write it back — three operations, not one. Two threads can both read the same old value, both add, and both write back the same result, so one update is silently lost. Make it atomic with `AtomicInteger.incrementAndGet`, a `synchronized` block / `Lock`, or `LongAdder` for hot counters.

2. 🌟 **"`volatile` vs `synchronized`?"** *(gotcha)* — `volatile` gives **visibility + ordering** (a write is immediately visible to later reads; it's a memory barrier so reads/writes aren't reordered across it) and makes single reads/writes of the field atomic *even for `long`/`double`* — but it does **not** make a compound action atomic, so `volatileCounter++` still loses updates. `synchronized` gives **mutual exclusion + visibility + ordering** across a whole block. Rule: `volatile` for a one-writer flag (e.g. a `stop` signal); `synchronized` when you need atomicity across multiple steps or fields.

3. **"What is happens-before, and what creates it?"** — A guarantee that one action's writes are **visible to** and **ordered before** another action. You never get it for free; specific constructs create the edges: monitor **unlock → lock** of the *same* monitor, **`volatile` write → read** of the same field, **`Thread.start()`** → the started thread's first action, a thread's actions → another thread's successful **`join()`**, and **`final`-field** publication of a correctly constructed object.

4. **"`AtomicLong` vs `synchronized` vs `LongAdder` — when each?"** *(applied)* — `AtomicLong` (lock-free CAS) for a single counter at low/medium contention — no blocking, just occasional retries. `LongAdder` for a *hot* counter at high contention — it stripes the count across cells so threads usually hit different memory (no CAS contention / false sharing), and `sum()` aggregates. `synchronized`/`Lock` when you must update **multiple fields** atomically (an invariant a single CAS can't express).

5. 🌟 **"What are virtual threads, and what do they change?"** *(version-evolution)* — Lightweight threads (stable Java 21, JEP 444) scheduled by the JVM onto a small pool of **carrier** platform threads. When a virtual thread **blocks** (I/O, `sleep`, most `j.u.c` waits) it **unmounts** the carrier, freeing it to run another — so blocking is cheap and you can run millions, writing simple blocking code instead of reactive callbacks. They do **not** change the memory model and do **not** make racy code safe; you no longer pool them (one thread per task).

6. **"How would you prevent two simultaneous withdrawals from overdrawing an account?"** *(concurrency + money — the Step 12 setup)* — Make check-and-debit **atomic**. Inside one JVM: a lock (`synchronized`/`ReentrantLock`) or an atomic compound op. Across instances (the real bank): push the atomicity to the database — a pessimistic row lock (`SELECT … FOR UPDATE`, Step 10), an optimistic `@Version` with retry (Step 9), or `SERIALIZABLE` + retry — or a distributed lock. In-JVM `synchronized` alone only works with a single instance.

> **Behavioral / STAR seed:** *"Tell me about a hard bug you debugged."* — A race that only failed under load. **S/T:** intermittent balance corruption in prod, green on the laptop. **A:** reproduce it *deterministically* (a `CyclicBarrier` that forces the interleaving) so it fails every run, fix with the right primitive, add a stress test. **R:** regression-proof, and a reusable pattern for the team.

## Step 12 — Demand Account, the Double-Entry Ledger & Transactions Deep

1. 🌟 **"Two withdrawals hit the same account at the same instant — what happens, and how do you make it correct?"** *(the money question)* — Without protection it's a lost-update / overdraft race (the balance is a read-check-write). Fix by making check-and-debit **atomic on the contended row**: a pessimistic row lock (`SELECT … FOR UPDATE`) to serialize, optimistic `@Version` + a retry loop, or `SERIALIZABLE` + retry. For a **hot** account, pessimistic; for **rare** conflicts, optimistic.

2. 🌟 **"Optimistic vs pessimistic locking — trade-offs?"** *(the most common)* — Optimistic (`@Version`): no lock during the read, detect the conflict at write time (the `UPDATE … WHERE version = ?` matches 0 rows → `ObjectOptimisticLockingFailureException`) and **retry**; best when conflicts are rare. Pessimistic (`SELECT … FOR UPDATE`): take a row lock at read time, others **block**; best under high contention when you must serialize. Measured here: 20/20 transfers succeed with `FOR UPDATE`; **17/20 fail** with only `@Version`.

3. **"What does `@Transactional(propagation = REQUIRES_NEW)` do, and when would you use it?"** — Suspends the caller's transaction and runs in a **brand-new** one that commits independently — so e.g. an **audit/log** row survives even when the business transaction rolls back. (The default `REQUIRED` joins the caller's transaction and rolls back with it.) It only works across a **bean boundary**: a `this.`-call is self-invocation and bypasses the proxy.

4. **"Does `@Transactional` roll back on a checked exception?"** *(gotcha)* — No. By default it rolls back only on `RuntimeException`/`Error`; **checked exceptions commit** unless you set `rollbackFor`. (That's why `InsufficientFundsException extends RuntimeException` — a failed transfer rolls back automatically.) Bonus gotcha: `@Transactional` is proxy-based, so a self-invocation (`this.method()`) starts no transaction at all.

5. 🌟 **"Why `BigDecimal` for money, not `double`?"** — `double`/`float` are binary floating point and can't represent decimals like `0.10` exactly, so cents drift over many operations. `BigDecimal` is exact decimal: store as `numeric(19,4)`, **compare with `compareTo`** (not `equals`, which is scale-sensitive: `100.00` ≠ `100.0000`), and round explicitly with a `RoundingMode`.

6. **"How do you avoid deadlocks when a transfer locks two rows?"** *(concurrency)* — Acquire the locks in a **consistent global order** (here, by account number) regardless of transfer direction, so A→B and B→A both lock the lower-numbered account first and can't form a hold-and-wait cycle. (Postgres also detects deadlocks and aborts one with `40P01`, but ordering avoids them entirely.)

7. **"Why a double-entry ledger, and how do you know the books balance?"** *(domain + design)* — Every movement is two equal entries — a DEBIT on the payer, a CREDIT on the payee — sharing one `transactionId`, on an **append-only** table. Because every debit has a matching credit, the sum of all entries (credits − debits) is always **zero**; assert `ledgerNet == 0` plus `totalSystemBalance` constant to prove correctness. The materialized `balance` is a fast-read projection kept in sync inside the same transaction (event sourcing in Step 52 makes the ledger the only source of truth).

> **Behavioral / STAR seed:** *"Tell me about a time you prevented a serious bug before production."* — The overdraft race. **S/T:** concurrent transfers could double-spend (a TOCTOU bug), green on a single thread. **A:** reproduced it *deterministically* with a `CyclicBarrier` stress test that lost an update every run, added pessimistic `SELECT … FOR UPDATE` with deadlock-safe lock ordering, and a money-conservation assertion. **R:** the test now fails-closed and the books always balance under 20-thread load.

## Step 13 — Spring MVC / REST Deep (Problem Details, OpenAPI, Filters vs Interceptors)

1. 🌟 **"Walk me through what happens when a request hits a Spring MVC controller."** *(the classic)* — Servlet **filter chain** → **`DispatcherServlet`** (front controller) → **`HandlerMapping`** finds the `@RequestMapping` method → interceptors' **`preHandle`** → **`HandlerAdapter`** binds the arguments (an `HttpMessageConverter` deserializes `@RequestBody`, `@Valid` triggers Bean Validation) and invokes your method → the return value is serialized by a converter (Jackson → JSON) → `postHandle`/`afterCompletion`. Anything thrown goes to the `HandlerExceptionResolver`s — which include your `@ControllerAdvice`.

2. 🌟 **"How do you do global error handling, and what is RFC 9457?"** *(very common)* — A `@RestControllerAdvice` (a `@ControllerAdvice` + `@ResponseBody`) with `@ExceptionHandler` methods that return a **`ProblemDetail`**. Extend **`ResponseEntityExceptionHandler`** so the framework's built-in MVC exceptions (validation, unreadable body, 404/405) also come back as Problem Details. **RFC 9457** (successor to 7807) standardizes the media type **`application/problem+json`** and the body — `type` (problem-kind URI), `title`, `status`, `detail`, `instance`, plus custom extension members — so clients parse errors reliably instead of string-matching bespoke JSON.

3. 🌟 **"Filter vs interceptor — when do you use each?"** *(gotcha)* — A **`Filter`** is at the **servlet-container** level (wraps the whole `DispatcherServlet`), sees **every** request (even ones that 404 with no handler), and has **no handler context** — use it for correlation ids, CORS, compression, request/response wrapping. A **`HandlerInterceptor`** is at the **Spring-MVC** level, runs around the **matched handler**, and **knows which handler** ran (`preHandle`/`postHandle`/`afterCompletion`) — use it for per-handler timing, auth that needs handler metadata, MDC setup. Key detail: **filters auto-register** (just be a bean); **interceptors must be registered** via `WebMvcConfigurer.addInterceptors(...)`.

4. **"Is a `@RestController` thread-safe? What about your filter/interceptor?"** *(concurrency — Step 11 callback)* — Controllers, `@ControllerAdvice`, filters, and interceptors are **singletons** shared across all request threads, so they must be **stateless**: no mutable instance fields holding per-request data (that's a race). Per-request state belongs in **request attributes** (`request.setAttribute(...)`, as our `TimingInterceptor` does), `ThreadLocal`/`MDC`, or method locals — never an instance field.

5. **"How is your OpenAPI/Swagger doc generated, and what changed across versions?"** *(version-evolution)* — **springdoc-openapi** introspects the `RequestMappingHandlerMapping`, request/response types, and validation annotations to emit **OpenAPI 3.1** at `/v3/api-docs` and serve **Swagger UI** at `/swagger-ui.html`; you only supply metadata via an `OpenAPI` bean. The old **springfox** is unmaintained and stuck on older Spring — don't use it. springdoc **3.0.x** supports Spring Boot 4 / Framework 7 (2.8.x targets Boot 3), so it's pinned (it's not in Boot's managed BOM).

6. **"How do you return the right status for a business error like insufficient funds?"** *(applied)* — Map the **domain exception** in the advice to a `ProblemDetail` with the right code (**422 Unprocessable Entity** — well-formed but can't be fulfilled — vs **400** for a malformed request), a stable `type` URI, and a **safe `detail`** (never a raw stack trace, SQL, or secret). Validation failures → 400 with a per-field `errors` map attached via `problem.setProperty("errors", ...)`.

> **Behavioral / STAR seed:** *"Tell me about a time you improved an API's developer experience."* — **S/T:** clients integrated against ad-hoc `{"error":...}` JSON and had to read source to learn the endpoints. **A:** replaced the error envelope with **RFC 9457 Problem Details** (machine-readable `type`/`status` + a per-field `errors` map) and added a generated **Swagger UI** + correlation-id header for support. **R:** client integration bugs dropped, onboarding sped up, and incidents became traceable by request id.

## Step 14 — API Design, Versioning, Idempotency & Webhooks

1. 🌟 **"How do you make a payment API safe to retry?"** *(the fintech classic)* — An **`Idempotency-Key`** header + a server-side store of `key → result`. The client sends a unique key per logical operation on every attempt; the server records the key→result the first time and, on any retry with that key, returns the **stored** result without re-executing. So a timeout-then-retry can't double-charge. Keys get a **TTL** (you don't remember them forever), and the store's **unique constraint** is the concurrency guard for racing duplicates.

2. 🌟 **"How would you secure an outbound webhook?"** *(security)* — Sign each delivery with **HMAC-SHA256** over `timestamp + "." + body` using a **per-partner shared secret**; send the signature + timestamp as headers. The receiver recomputes the HMAC and compares in **constant time** (`MessageDigest.isEqual` — never `equals`, which leaks timing), and **rejects timestamps outside a tolerance window** (replay protection). Deliver over HTTPS and rotate secrets. This is the Stripe/GitHub scheme. HMAC (not a plain `hash(secret∥body)`) avoids length-extension attacks.

3. **"Webhook delivery semantics — and the consequence?"** *(gotcha)* — **At-least-once**: you retry on failure, so a receiver may see the **same event twice** → receivers **must be idempotent** (dedupe by event id). True exactly-once *delivery* is impossible in general; you get exactly-once **effect** via idempotent receivers, plus the **Outbox pattern** (Step 20) to avoid lost/dual writes.

4. **"How do you version an API without breaking clients?"** *(applied)* — Pick a strategy — **URI** (`/api/v1`, visible/cacheable/curl-able — chosen here), **header** (`Accept: …v1+json`, clean URLs but invisible), or **media-type** (most RESTful, heaviest). Add new versions **additively**; when retiring an old endpoint, keep it working and **deprecate gracefully** with `Deprecation: true`, a `Sunset` removal date, and a `Link; rel="successor-version"` (RFC 8594) — never a flag-day removal.

5. **"Offset vs cursor pagination?"** — **Offset** (`page`/`size`) is simple and supports random page access, but degrades at depth (`OFFSET 1_000_000` scans and discards a million rows) and can skip/duplicate rows if data changes between pages. **Cursor/keyset** ("give me 20 after id X / createdAt T") is stable and fast at any depth, at the cost of no random access. Either way, return a **stable envelope you own** (a `PageResponse` DTO), never the ORM's `Page` (its JSON shape is an unstable internal detail Spring warns against exposing).

6. 🌟 **"Two retries of the same idempotent transfer race — what happens?"** *(concurrency)* — Both may **miss** the key lookup and both execute the transfer, but the idempotency key's **PRIMARY KEY** lets only one commit the key row; the other's `INSERT` violates the unique constraint and its **whole transaction rolls back** (undoing its transfer too). Exactly one transfer commits. The database is the coordination point — the same principle as Step 12's `SELECT … FOR UPDATE`.

7. **"Why constant-time signature comparison?"** *(gotcha)* — A normal `equals`/`==` returns as soon as it hits a mismatched byte, so the response time leaks how many leading bytes matched — letting an attacker brute-force a valid signature byte by byte. `MessageDigest.isEqual` examines all bytes regardless of where the first mismatch is, removing the timing signal.

> **Behavioral / STAR seed:** *"Tell me about a time you prevented a costly bug."* — **S/T:** the transfer API's `POST` wasn't idempotent, so a client retry after a timeout could **double-charge**. **A:** added a Stripe-style `Idempotency-Key` + a key store (PK = the concurrency guard), proved with a test that a same-key retry moves money **once**, and HMAC-signed the outbound webhooks (with a replay window) so partners couldn't be spoofed or replayed. **R:** money-moving became safe to retry under both sequential and concurrent duplicates, all test-proven, with the dual-write gap documented for the Outbox follow-up.

## Step 15 — API Gateway / BFF & Service-to-Service HTTP

1. 🌟 **"Draw a microservices architecture — where's the gateway and what does it do?"** *(system design — the opener)* — A single **front door** every external client hits; it **routes** each request to the right internal service and centralizes **edge concerns** (authentication/authorization, rate limiting, TLS termination, correlation-id origination, request/response shaping) so each service doesn't reinvent them. It keeps internal services **private** and decouples clients from the topology (services can split/merge behind it). What does **not** belong in it: **business logic** — keep the gateway a thin edge. A **BFF** is the same idea specialized per client (web/mobile), often aggregating several service calls into one tailored response.

2. 🌟 **"How do services call each other, and what's the modern Spring way?"** — Two styles: **synchronous** over HTTP (request/reply — immediate but *temporally coupled*) or **asynchronous** via messaging (Kafka — decoupled, eventually consistent). For sync in Spring today: a **declarative HTTP interface** — an `@HttpExchange`/`@GetExchange` interface whose implementation is generated by `HttpServiceProxyFactory` over a `RestClient`. Type-safe, no hand-written plumbing, no extra dependency (it's in Spring Framework 6.1+/7) — the successor to `RestTemplate` (maintenance) and OpenFeign (for in-Spring use).

3. 🌟 **"What happens when a downstream service is slow?"** *(the resilience gotcha)* — Without a **timeout**, the caller's thread blocks waiting; a chain of such blocked calls **exhausts the thread/connection pool** and the slowness **cascades** into a system-wide outage. The baseline fix is **connect + read timeouts** so a slow dependency **fails fast** (Spring surfaces it as `ResourceAccessException`) and frees the thread. A connect timeout **alone** isn't enough — a downstream that accepts the connection then stalls still hangs you, so you need the **read** timeout too. The next layers — **circuit breakers, bulkheads, retries with backoff, fallbacks** — are Resilience4j (Step 37).

4. **"Reactive vs servlet Spring Cloud Gateway — which and why?"** *(version/architecture)* — The **reactive** gateway (`-server-webflux`, WebFlux/Netty) scales connections on an event loop — great for very high fan-out/streaming, but a different programming model. The **servlet** gateway (`-server-webmvc`, Spring MVC + a blocking HTTP client) fits an MVC stack and, with **virtual threads**, scales blocking forwards without the reactive complexity. We use **servlet** because the platform is MVC + virtual threads. (Config prefix: `spring.cloud.gateway.server.webmvc.routes` since Spring Cloud 2025; the old `spring.cloud.gateway.mvc.routes` is deprecated.)

5. **"Sync HTTP vs async messaging — when each?"** — **Sync** when the caller needs an immediate answer and can't proceed without it (and you accept temporal coupling). **Async** when you want decoupling/resilience and can tolerate eventual consistency — events, the **Outbox** pattern, and **Saga** (Steps 20–21). Most real systems use both: sync for north-south request/reply, async for east-west workflows.

6. **"How would you secure the edge?"** *(security)* — Authenticate/authorize **at the gateway** (Steps 16–17), **rate-limit** (Step 51), **terminate TLS**, **strip spoofable inbound headers** (so clients can't forge gateway-added identity/correlation headers), and keep internal services **unreachable from outside** (NetworkPolicy, Step 42). The gateway is the perimeter; also gate `/actuator/gateway/routes` so you don't leak your internal topology.

7. **"What does `StripPrefix` do, and why route by a service prefix?"** *(applied)* — A routing scheme: external `/cif/**` and `/bank/**` carry a **service prefix** that selects the route; `StripPrefix=1` removes that leading segment before forwarding, so the downstream receives its **own** path (`/cif/api/customers/1` → cif's `/api/customers/1`). It gives clients one address per logical area while each service keeps its native API.

> **Behavioral / STAR seed:** *"Tell me about a time you prevented a cascading failure."* — **S/T:** an inter-service HTTP call had **no read timeout**, so a slow dependency could tie up caller threads until exhaustion. **A:** added connect + read timeouts (fail fast → `ResourceAccessException`) on the declarative client, proved it with a test where a 1.5 s stub trips a 400 ms timeout, and flagged circuit breakers/bulkheads as the next layer (Resilience4j, Step 37). **R:** a downstream slowdown stopped propagating into a caller-wide outage, with the resilience roadmap documented.

## Step 16 — Spring Security Deep I (Filter Chain, JWT & Password Encoding)

1. 🌟 **"How does Spring Security work — what's the filter chain?"** *(the classic opener)* — At heart it's **a chain of servlet filters** placed in front of your app (one `FilterChainProxy` delegating to an ordered `SecurityFilterChain`). Every request runs the chain *before* the `DispatcherServlet`/controller. Two jobs, in order: **authentication** (establish identity — here decode+validate the Bearer JWT; none/invalid → **401** via the `AuthenticationEntryPoint`) then **authorization** (apply access rules; not allowed → **403** via the `AccessDeniedHandler`). You configure it by declaring a **`SecurityFilterChain` bean** built from `HttpSecurity` with the lambda DSL — the modern replacement for `WebSecurityConfigurerAdapter`, which was **removed in Spring Security 6**.

2. 🌟 **"401 vs 403 — what's the difference?"** *(the gotcha)* — **401 Unauthorized** = *not authenticated*: no credentials, or invalid/expired ones (the request couldn't establish who you are). **403 Forbidden** = *authenticated but not authorized*: we know who you are, but you lack the required role/permission. They surface at different points of the chain (`AuthenticationEntryPoint` vs `AccessDeniedHandler`). Watch the leakage angle too: a bare 403 can reveal a resource exists — keep responses generic.

3. 🌟 **"What's a JWT and how do you validate it?"** — A **JSON Web Token**: three base64url parts — `header.claims.signature`. The header names the algorithm; the claims carry `sub` (subject), `roles`, `iss`, `iat`, `exp`; the signature (HMAC-SHA256 here) lets a key-holder verify it wasn't forged/tampered. To validate: check the **signature** (with the shared secret or the issuer's public key) **and** the **expiry**, then map claims → authorities. As an **OAuth2 resource server**, Spring's `BearerTokenAuthenticationFilter` + `JwtDecoder` do this per request — **stateless**, no session. A JWT is **signed, not encrypted** — the payload is readable, so never put secrets/PII in claims.

4. **"Symmetric vs asymmetric JWT signing — which and when?"** *(architecture/version)* — **HMAC (HS256)**: one shared secret signs *and* validates. Simple and self-contained for **one issuer + one validator** — but every validator holding the secret can also **forge** tokens. **Asymmetric (RSA/EC)**: the issuer signs with a **private** key; each service validates with the **public** key (fetched from a **JWKS** endpoint, `/.well-known/jwks.json`) and *cannot* mint tokens — least privilege. Move to asymmetric when **many independent services validate** (Step 17/41). We use HMAC now (one auth service) and pin the algorithm on the decoder.

5. 🌟 **"How do you store passwords?"** — Never plaintext, never a fast unsalted hash (MD5/SHA-1 — crackable at scale, rainbow tables). Use a **slow, salted, one-way** hash: **BCrypt** (`$2a$10$<salt><hash>` — cost factor + per-hash random salt baked in), or Argon2/scrypt. Verify with `PasswordEncoder.matches(raw, hash)` (constant-time, never `equals`). The same password hashes **differently** each time (distinct salts) yet both verify — that's what defeats rainbow tables. Tune the cost factor up as hardware improves.

6. **"Do you need CSRF protection for a JWT API?"** — **No** for a **stateless Bearer-token** API: CSRF attacks ride **ambient** credentials (cookies/session the browser sends automatically), and a token API has none — the client must *explicitly* attach the token, so there's nothing to forge (and leaving CSRF on would block non-browser clients). **Yes** for **cookie/session** flows. The rule: *no ambient credentials → no CSRF risk* — and know which mode you're in, because disabling CSRF on a cookie app is a real vulnerability.

7. **"Why is there no auth code inside your `/admin` controller method?"** *(applied)* — Because the **filter chain** already enforced `requestMatchers("/api/auth/admin").hasRole("ADMIN")` *before* the method runs — a non-admin got a 403 from the `AuthorizationFilter` and never reached the controller. Authorization lives in **one** place (the chain, or method security via `@PreAuthorize` — not duplicated ad-hoc in the body), so the rule can't drift out of sync. The controller just *reads* the identity (`Authentication` injected from the `SecurityContextHolder`), it doesn't *establish* it.

> **Behavioral / STAR seed:** *"Tell me about adding security to a system."* — **S/T:** the bank's services were entirely unauthenticated; we needed identity that scales across microservices without server sessions. **A:** stood up a dedicated `services/auth` with a stateless `SecurityFilterChain` (CSRF off — Bearer tokens, no ambient creds), BCrypt-hashed passwords, and short-lived HS256 JWTs issued on login and validated as an OAuth2 resource server; mapped a `roles` claim to authorities for `hasRole` checks. **R:** proved 401 (no token) / 403 (wrong role) / 200 (authorized) over real HTTP plus a §12.3 mutation that breaks the 403 test, and documented the next moves — asymmetric JWKS signing + refresh tokens (ADR-0008, Step 17).

## Step 17 — Spring Security Deep II (Resource Servers, RS256/JWKS, Method Security & Modern Auth)

1. 🌟 **"How do multiple services validate the same JWTs without sharing a secret?"** *(the system-design opener)* — **Asymmetric signing + JWKS.** The issuer (auth) signs with a **private** RSA key (RS256) and publishes the matching **public** key as a **JWKS** (JSON Web Key Set) at a well-known URL (`/oauth2/jwks`). Each service fetches and **caches** the public key (Spring's `jwk-set-uri`) and validates token signatures **locally** — no shared secret to leak, no per-request call back to auth. Tokens carry a `kid` so verifiers pick the right key, which makes **rotation** seamless. This is how OAuth2/OIDC providers (Google, Auth0, Keycloak) work.

2. **"HMAC vs RSA (HS256 vs RS256) for JWT — when each?"** *(gotcha)* — **HMAC (HS256)**: one shared secret signs *and* validates — simplest, but every validator can also **forge**, so it's only safe within a **single** service. **RSA/EC (RS256/ES256, asymmetric)**: the issuer signs with the **private** key; many services validate with the **public** key (via JWKS) and *cannot* mint tokens — least privilege. Switch to asymmetric the moment **more than one independent service** validates the same tokens.

3. 🌟 **"What is an OAuth2 resource server?"** — A service that authorizes requests by validating a **Bearer access token** (signature + expiry + scopes/claims) on **each request, statelessly**, against a configured issuer — no login UI, no session, no cookies. In Spring you set `spring.security.oauth2.resourceserver.jwt.jwk-set-uri` and `http.oauth2ResourceServer(oauth2 -> oauth2.jwt(...))`; Boot wires a `BearerTokenAuthenticationFilter` + a `JwtDecoder` (built from the JWKS) that runs before your controller. demand-account became one this step.

4. **"URL authorization vs method security — when each, and what's the catch?"** — **URL rules** (`authorizeHttpRequests().requestMatchers(...)`) gate the **HTTP edge** coarsely (e.g. "all `/api/**` need a token"). **Method security** (`@EnableMethodSecurity` + `@PreAuthorize("hasRole('ADMIN')")`) expresses **fine-grained, domain-level, reusable** rules right on a method. Real systems use **both**. *The catch:* method security wraps the bean in an **AOP proxy** and applies to **direct in-process calls too** — so annotating a *service* method forces a `SecurityContext` into every unit test that calls it directly. We deliberately kept authz at the edge + a narrow `@PreAuthorize` demo to avoid that blast radius (ADR-0009).

5. **"How do you revoke a JWT?"** *(gotcha)* — Mostly **you can't** before it expires — a self-contained JWT carries no server-side lookup. Mitigations: **short-lived** access tokens (minutes) + a longer-lived, **revocable refresh token** to mint new ones; a **`jti` denylist** (reintroduces some state); and **key rotation** via JWKS (rotating the signing key invalidates everything signed with the old one). If you need *instant* revocation, use **opaque tokens + introspection** (a per-request call to the issuer) instead.

6. **"What's the algorithm-confusion attack, and how does Spring defend?"** *(security gotcha)* — In an RS256 deployment the verification key is **public**. An attacker crafts an **HS256** token using that *public key as the HMAC secret*; a naive verifier that trusts the token's own `alg` header would validate it. Defense: **pin/derive the expected algorithm from the key, never from the token.** Spring's `NimbusJwtDecoder` built from a JWKS validates against the RSA key type, so an HS256 token simply doesn't match — and you never accept `alg: none`.

7. **"What are passkeys, MFA, and step-up auth?"** *(modern)* — **MFA**: a second factor (something you *have*/*are*) beyond the password. **Passkeys/WebAuthn**: phishing-resistant, **device-bound public-key** credentials — the device's biometric unlocks a private key that signs a server challenge; nothing phishable leaves the device. **Step-up**: requiring a **stronger/fresh** factor for a sensitive action (e.g. a large transfer) even though you're already logged in. Spring Security 7 models *how* you authenticated as `FACTOR_*` authorities, so a `@PreAuthorize("hasAuthority('FACTOR_OTP')")` can demand step-up.

> **Behavioral / STAR seed:** *"Tell me about securing a distributed system."* — **S/T:** the bank had one auth service signing HMAC JWTs, but a second service (money movement) now had to trust them — and a shared secret would let any service **forge** tokens. **A:** moved auth to **asymmetric RS256** signing + a published **JWKS**, made demand-account an **OAuth2 resource server** validating via `jwk-set-uri` (public key only), and added URL + `@PreAuthorize` authorization (with a deliberate edge-vs-service blast-radius choice, ADR-0009). **R:** proved a **live cross-service flow** — the money service fetched auth's JWKS, accepted an auth-minted token (201), rejected an unauthenticated request (401), and enforced ADMIN (403/200) — cryptographic trust across services with **no shared secret**, plus a §12.3 mutation showing the auth rule is load-bearing.

## Step 18 — Secure Coding & Threat Modeling (STRIDE, OWASP, DevSecOps Shift-Left)

1. 🌟 **"What is BOLA / IDOR, and how do you prevent it?"** *(the #1 API-security question)* — **Broken Object Level Authorization** (OWASP API1:2023; classically *IDOR*, Insecure Direct Object Reference): an endpoint takes an object identifier from the request (`{"from":"ACC-A"}`, `/accounts/{id}`) and acts on it **without checking the authenticated caller is entitled to that object**. Authentication passes — the failure is *authorization at the object level*. Prevention: give objects an **owner** (set server-side at creation from the principal, e.g. `jwt.getSubject()`), and on **every** read/write path compare caller → owner (else 403, with an explicit admin override). Never trust an id in the request to imply entitlement. It's #1 because it needs no exploit — just a valid login and someone else's id — and generic tooling can't find it (the request looks perfectly well-formed).

2. **"Walk me through how you'd threat-model a system."** — Shostack's four questions: **① What are we building?** Draw a **DFD** (external entities, processes, data stores, flows) and mark the **trust boundaries** — every crossing must authenticate, authorize, validate. **② What can go wrong?** Run **STRIDE per element** (Spoofing, Tampering, Repudiation, Information disclosure, DoS, Elevation) over every box and arrow, recording ✅/🟡/🔴 with evidence; add **attack trees** for the crown-jewel goals ("steal money") to test reachability in depth. **③ What do we do about it?** Ship the cheap complete fixes now; everything else becomes a **risk register** entry — ID, severity, mapped category, *specified* fix, owner, ETA. **④ Did we do a good job?** Re-review on a concrete trigger (new service, new boundary). In our bank this found one CRITICAL (BOLA) plus five tracked gaps, and drove tested header/CORS hardening the same step.

3. 🌟 **"How do you *really* stop SQL injection?"** — **Parameterized queries / prepared statements** — never escaping, never blocklists. Over the wire (e.g. Postgres's extended query protocol) the SQL text is **parsed into a plan before the values are bound**, so user input physically cannot alter the statement's structure: it's always *data*, never *code*. ORMs like Spring Data do this for you (`findByCustomerNumber` → `WHERE customer_number = ?`); the danger zone is hand-concatenated native SQL. We *prove* it with a contrast test: the same `' OR '1'='1` payload matches **0** rows through the parameterized repository and **every** row through a deliberately concatenated query — 0 vs 1 is the whole lesson, and it shows the test has teeth.

4. **"Is CORS a security control? Configure it correctly for an API."** *(gotcha)* — CORS is a **browser guardrail, not an access control**: only browsers honor the preflight; curl and other servers ignore it entirely, and server-to-server callers send no `Origin` header at all. So it cannot replace authn/authz — but a **too-permissive** policy (`*` or reflected origins, especially with credentials) actively donates your users' sessions to any malicious page. Correct shape: **deny-by-default** — an explicit allow-list (empty until a front-end exists), explicit methods and headers (don't forget custom ones like `Idempotency-Key`), a preflight `max-age`. In Spring: a `CorsConfigurationSource` **bean named `corsConfigurationSource`** + `.cors(Customizer.withDefaults())` — Spring Security resolves it *by name* (injecting it by type fails: MVC's `mvcHandlerMappingIntrospector` also implements the interface → two beans).

5. **"STRIDE the login flow for me."** *(applied)* — **S**: brute-force/credential-stuffing → BCrypt (slow+salted) + lockout/rate limit; **T**: forge or alter the token → asymmetric RS256 signing, algorithm pinned to the key (kills `alg:none`/confusion); **R**: "I never logged in" → audit events with `sub`/`iat`; **I**: leak hashes or the private key → hashes never returned, JWKS publishes the public half only; **D**: login flood → throttle the endpoint; **E**: self-granted roles → roles assigned server-side from the user store, never read from client input.

6. **"What's a trust boundary, and why do threat models obsess over them?"** — The line where data moves between trust levels (internet→edge, edge→internal, service→DB, you→partner). Threats live on **crossings**: each one is a checkpoint that must authenticate, authorize, and validate, so enumerating boundaries turns "think about security" into a *finite checklist of arrows*. Most real findings are a crossing where one of the three checks was assumed instead of enforced — ours was edge→internal, where authentication existed but object-level authorization didn't.

7. **"Name the security headers you'd ship by default and what each does."** — `X-Content-Type-Options: nosniff` (browser must trust the declared Content-Type — blocks MIME-sniffing an upload into executable HTML); `X-Frame-Options: DENY` (no framing — kills clickjacking overlays); `Referrer-Policy: no-referrer` (outbound clicks don't leak your URLs); `Strict-Transport-Security` (HSTS — browser pins HTTPS for a year; only emitted over TLS, so don't assert it in plain-HTTP tests); plus `X-XSS-Protection: 0` (yes, **0** — disables the legacy, itself-exploitable XSS auditor) and no-store cache headers on API responses. In Spring Security these are `HeaderWriterFilter` writers configured via `.headers(...)`, stamped on **every** response including 401s.

> **Behavioral / STAR seed:** *"Tell me about a time you found a serious security issue."* — **S/T:** a STRIDE pass over our payment platform's transfer endpoint showed it accepted the source-account id from the request body with no ownership check — any authenticated user could move anyone's money (BOLA, API1:2023). **A:** instead of rushing a half-fix, I recorded it as a CRITICAL risk-register entry with a fully specified remediation (ownership column set from the JWT at creation, enforcement on every money/read path, owner/non-owner/admin tests) and an owner; documented the interim control (authentication still required — not anonymously exploitable); and shipped the cheap complete hardening the same model demanded — security headers and deny-by-default CORS, each behind a test, with a mutation check (`allowedOrigins("*")` → the deny test fails) proving the tests bite. **R:** the risk was visible and prioritized instead of latent; the hardening landed tested within days; the eventual ownership fix shipped *complete* with its own test matrix — and no one ever had false confidence about the gap in between.

## Step 19 — Distributed-Systems Theory & Trade-offs (CAP/PACELC, Consistency, Quorums, Clocks, Delivery)

1. 🌟 **"Explain the CAP theorem."** *(the single most-asked distributed-systems question)* — During a **network partition** (replicas can't communicate), a distributed data store can preserve either **Consistency** (every read sees the latest write — formally *linearizability*, not ACID's "C") or **Availability** (every request gets a non-error response) — **not both**. Two nuances interviewers probe: CAP only constrains behavior *during* a partition (it says nothing about normal operation), and "CA" is not a real option — partitions *will* happen (cable cuts, GC pauses, switch reboots), so the genuine choice is **CP** (refuse to serve rather than diverge — ZooKeeper, etcd) vs **AP** (keep serving, reconcile later — Dynamo-style stores). Strong answer ends with placement: a bank's *money* path is CP-flavored (correctness over uptime); a "last seen balance" widget can be AP.

2. **"What does PACELC add to CAP?"** — The half CAP omits: **if P**artition, choose **A** or **C**; **E**lse (healthy network), you *still* choose **L**atency or **C**onsistency. Keeping replicas in sync costs round-trips — wait for replicas (consistent, slower → EC) or ack locally and replicate asynchronously (fast, possibly stale → EL). Stores get classified like **PC/EC** (always consistent: Spanner-ish) or **PA/EL** (Dynamo: available + fast). The else-branch is why you can meet a stale read with *zero* failures anywhere — e.g. reading a lagging async PostgreSQL replica.

3. **"Order the consistency models and give an example of each."** — Strongest → weakest: **linearizable** (as if one copy, real-time order — an etcd compare-and-set), **sequential** (one global order, not necessarily real-time), **causal** (cause→effect order preserved; concurrent ops may be seen differently — what vector clocks track), **read-your-writes** (a session sees its own writes), **eventual** (replicas converge once writes stop — a Dynamo shopping cart). Stronger = more coordination = slower/less available; the model is a *budget decision*, not a virtue contest.

4. **"How do quorums give strong consistency?"** *(applied)* — With `N` replicas, write quorum `W`, read quorum `R`: if **`W + R > N`** then every read set must share at least one replica with every write set (pigeonhole), so a read always sees the latest committed write; adding **`W > N/2`** prevents two conflicting writes from both committing. The inequality must be *strict* — `N=5, W=3, R=2` gives `5 > 5 = false`: write `{0,1,2}`, read `{3,4}`, stale. Slackening to `W+R ≤ N` is a legitimate *choice* — cheaper, faster, eventually-consistent reads. It's a dial, not a bug (and our lab brute-forces all 25 (W,R) pairs for N=5 to prove the rule empirically).

5. **"Lamport clock vs vector clock?"** *(gotcha)* — **Lamport:** one counter per process; tick on local events, `max(local, received)+1` on receive. Guarantees `a → b ⇒ L(a) < L(b)` — but the **converse fails**: concurrent events still get ordered-looking numbers, so it *cannot detect concurrency*. **Vector:** one counter *per process*, carried as a vector; componentwise max on receive. `a → b` iff a's vector ≤ b's everywhere and < somewhere; incomparable vectors = **concurrent** (`a ∥ b`). Price: O(N) space per stamp. Use Lamport when you just need a causal order; vectors when you must *detect conflicts* (Dynamo siblings, causal consistency).

6. 🌟 **"Is exactly-once delivery possible?"** *(gotcha favorite)* — **Not delivery.** An asynchronous, failure-prone network offers **at-most-once** (may lose) or **at-least-once** (may duplicate) — when an ack goes missing, the sender must either not resend (risking loss) or resend (risking a duplicate); it cannot know which happened. What real systems (Kafka included) achieve is **exactly-once *effect***: tolerate at-least-once delivery and make the consumer **idempotent** — dedupe by a stable message id (`Set.add` semantics / a DB unique constraint / Step 14's Idempotency-Key) — or transactional. Bonus point: dedupe by **id, never by payload** — two legitimate identical-amount deposits are distinct transfers.

7. **"Your idempotent consumer keeps a 'seen ids' store — what are the concurrency and retention concerns?"** *(concurrency tie-in, Step 11 callback)* — The dedupe store is **shared mutable state across deliveries**: with concurrent consumers, check-and-record must be atomic (a `ConcurrentHashMap.newKeySet()` in-process; in production a **DB unique-constraint insert**, which makes the race the database's problem — same lost-update lesson as Step 11). Retention: an unbounded set grows forever; bound it with a TTL **at least as long as the producer's maximum retry window** (mirror of Step 14's webhook replay window), or duplicates older than the TTL resurrect.

> **Behavioral / STAR seed:** *"Tell me about a time you had to reason about a distributed-systems trade-off."* — **S/T:** our banking platform was about to go event-driven (Kafka, Sagas), and design reviews kept stalling on "make it consistent *and* always available." **A:** I built a small deterministic lab proving the trade-offs executable — a CP register that visibly refuses writes during a partition, an AP one that diverges and LWW-converges on heal, a brute-forced `W+R>N` quorum check, and a delivery sim contrasting naive (×3 = money invented) vs idempotent (exactly-once effect) consumers — and used it to anchor the team's vocabulary. **R:** we classified each data path explicitly (money = CP-flavored, balance widgets = AP-tolerable), standardized idempotent consumers before the first Kafka topic landed, and the "is exactly-once real?" debate ended with a test instead of opinions.

## Step 20 — Spring Events + Kafka, the Outbox Pattern & Real-Time Push (SSE)

1. 🌟 **"What's the dual-write problem and how does the Outbox pattern solve it?"** *(the most commonly asked)* — You cannot atomically "write the database AND publish to a broker": two systems, no shared transaction, so a crash between the two either loses an event (write-then-publish) or invents one for a change that never committed (publish-then-write). The **transactional Outbox** writes the event to an `outbox_event` table **in the same DB transaction** as the business change — both commit or neither — then a separate **relay** publishes unpublished rows to Kafka and marks each only **after** the broker's ack → at-least-once, never lost. Strong answer names the residual window (send acked, crash before the mark → duplicate on the topic) and its partner fix: an idempotent consumer.

2. **"`@EventListener` vs `@TransactionalEventListener`?"** — A plain `@EventListener` fires immediately at publish time — even if the surrounding transaction later rolls back (dangerous for money: you'd react to a transfer that never happened). `@TransactionalEventListener` parks the event in a `TransactionSynchronization` and fires at a chosen phase — `AFTER_COMMIT` (default), `BEFORE_COMMIT`, `AFTER_ROLLBACK`, `AFTER_COMPLETION` — so you only react to committed work. Gotcha follow-ups: with no active transaction the event is delivered immediately; and the listener runs *after* commit, so its own exception can't roll the work back.

3. 🌟 **"How do you get exactly-once with Kafka?"** *(gotcha favorite)* — Not exactly-once **delivery** — that's impossible over a failure-prone network (Step 19). Kafka's default contract is at-least-once: rebalances and producer retries can redeliver. The achievable goal is exactly-once **effect**: dedupe in the consumer by a stable event id (`Set.add` semantics in-process; a DB unique-constraint insert or Redis `SETNX` in production) so duplicates are no-ops. (Kafka's own "EOS" — idempotent producers + transactions — covers Kafka-to-Kafka topologies, not your DB-touching consumer.) Our proof: 3 deliveries carrying 2 distinct eventIds → received 3, applied exactly 2.

4. **"Why isn't publishing to Kafka from an AFTER_COMMIT listener reliable enough?"** — Commit succeeds → process dies before the listener's publish executes → the DB says it happened, the broker never hears it, and **nothing durable remembers the intent**, so no retry is possible: the event is lost forever. The Outbox closes exactly that gap by making the publish-intent a row inside the commit. This question is the #1 filter for "has actually built event-driven systems."

5. **"SSE vs WebSocket — when do you use which?"** — **SSE**: one-way server→client over a long-lived plain-HTTP response (`text/event-stream`); the browser's `EventSource` auto-reconnects; proxy/firewall friendly; in Spring MVC it's an `SseEmitter` riding servlet async (the request thread returns to the pool; any thread can later push frames). Ideal for notifications/feeds/progress. **WebSocket**: a bidirectional protocol upgrade — needed only when the *client* streams too (chat, collaborative editing, order entry). Don't pay for duplex you don't use.

6. **"Where does your event id come from, and why does it matter?"** *(applied)* — Minted once at the source (`UUID.randomUUID()` when the transfer emits its event) and carried unchanged end-to-end: outbox row **primary key** → Kafka record **key** (also in the payload) → consumer **dedupe key** → the field the UI shows. One identity across five systems makes "have I seen this before?" answerable at every hop, and makes the pipeline traceable. Dedupe by **id, never payload** — two identical $10 transfers are distinct events.

7. **(Concurrency, Step-11 callback)** **"Your Kafka listener thread and HTTP request threads share state — what, and how is it safe?"** — Three structures, three deliberately different tools: the dedupe set is `ConcurrentHashMap.newKeySet()` (the `add` return value makes check-and-record a single atomic op — no check-then-act race); the SSE emitter list is `CopyOnWriteArrayList` (broadcast iteration never throws `ConcurrentModificationException` while clients connect/disconnect); the recent-notifications deque sits under `synchronized` because `addFirst` + trim-to-50 must be atomic *together* — a compound action needs a lock, not just a concurrent collection.

> **Behavioral / STAR seed:** *"Tell me about a time you made an unreliable integration reliable."* — **S/T:** our banking platform's transfers had to notify downstream consumers, and the naive save-then-publish design silently lost events whenever a node died between the DB commit and the broker publish. **A:** I implemented the transactional Outbox (event row commits atomically with the ledger write), a polling relay that marks rows published only after the broker's ack (at-least-once by construction), and an idempotent consumer deduping by event id — then proved both halves against a real broker in CI (Testcontainers Redpanda), including a mutation run where disabling the dedupe makes the exactly-once test fail (`expected: 2 but was: 3`). **R:** demonstrably zero lost events (we stopped the broker mid-demo; the event arrived after restart), exactly-once effect under duplicates, a live SSE surface product could build on — and a documented upgrade path to CDC/Debezium instead of polling.

## Step 21 — Payments: the Saga Pattern, Redis Idempotency Keys & Dead-Letter Topics

1. 🌟 **"What is a Saga and a compensating transaction?"** *(the most commonly asked)* — A long-running operation split into local transactions that each commit independently; if a later step fails, you run **compensating** transactions that semantically undo the committed ones (a refund undoes a debit — a new, opposite action, not a rollback; once a step commits, nothing can un-commit it). Used when no single ACID transaction spans the steps (cross-service, cross-shard, external systems). Strong answer adds: a failed-but-compensated Saga leaves *history* (our ledger shows the DEBIT plus a refund CREDIT netting to zero — append-only books), and compensation itself must be durable and retryable, because it can fail too.

2. **"Orchestration vs choreography?"** — Orchestration: a central coordinator drives the steps and decides compensations — explicit flow, one place to test and trace, risk of a god-class. Choreography: services react to each other's events with no coordinator — looser coupling, but the flow is implicit across services and debugging means correlating event streams. Rule of thumb: orchestrate short critical flows (payments); choreograph broad, loosely-coupled reactions (notifications, analytics). Ours is orchestration: `PaymentService` calls `debit` → `credit` and runs `refund` on failure.

3. **"How do you make a payment idempotent?"** — A client-supplied **Idempotency-Key**: atomically record key→result on first execution — Redis `SET key value NX EX ttl` (`NX` makes check-and-claim one atomic server-side command; `EX` bounds the key to the retry window) — and return the stored result for any retry with the same key, so the client can't tell a replay from the original. Volunteer the gotchas: record only on *success* (a failed attempt must stay retryable), and the naive check-execute-record flow still has a concurrent-duplicate window — reserve-then-complete (claim `PENDING` first) closes it.

4. **"What's a Dead-Letter Topic and why do you need one?"** — A topic where the consumer's error handler routes messages that still fail after bounded retries, so a poison message can't block the partition forever (Kafka can't advance past a failing offset) and isn't silently dropped either — it's preserved verbatim (plus diagnostic headers: original topic/partition/offset, exception) for inspection and re-drive. In Spring Kafka: `DefaultErrorHandler` (seek-back retries, e.g. `FixedBackOff(0, 2)` = 3 total attempts) + `DeadLetterPublishingRecoverer` publishing to `<topic>.DLT`. The subtle prerequisite: the listener must *throw* — a swallow-all `try/catch` makes the whole mechanism permanently idle.

5. **"Saga vs two-phase commit (2PC)?"** — 2PC gives atomicity across participants but a coordinator holds everyone's locks while collecting votes: slow, availability-hostile (a crashed coordinator leaves participants *in doubt*, locks held), and unsupported by Kafka and most HTTP services. The Saga is the modern default: available and eventually consistent, paying with explicit compensation logic and visible intermediate states. Bonus: 2PC is consensus-shaped machinery in your request path — modern designs reserve real consensus for infrastructure (the broker's own Raft) and use Sagas + outboxes + idempotency at the application layer.

6. **(Gotcha) "Is a Saga isolated?"** — No. Between steps the intermediate state is real and visible — money has left A and not yet reached B, and any reader (including your own API) sees the debited balance mid-Saga. ACID's "I" is the letter a Saga doesn't have. You design for it: pending/in-flight states, idempotent steps, and never deriving invariants from mid-Saga reads.

7. **(Concurrency, Step-11 callback) "Two threads submit the same Idempotency-Key at the same instant — what happens?"** — The DB row lock (`SELECT … FOR UPDATE`) keeps each money leg correct regardless. But both threads can pass the Redis *check* before either *records* — a check-then-act race (Step 11) stretched across a process boundary — so both may run the Saga: double payment, two ids; `SET NX` then keeps only the first recorded result. Fix: make the *claim* the first action (reserve-then-complete, `setIfAbsent(key, "PENDING", ttl)` and only the winner proceeds) so the race collapses onto Redis's single-threaded atomic `SET NX` — the same "make the store the arbiter" move as Step 14's unique-constraint insert.

> **Behavioral / STAR seed:** *"Tell me about a time you designed for partial failure."* — **S/T:** our banking platform needed cross-account payments that couldn't ride a single transaction, and the failure mode that scared us was money leaving the source and never arriving. **A:** I modeled the payment as an orchestrated Saga — each leg an independently-committed transaction with a row lock, an explicit refund compensation, and a typed `PaymentFailedException` mapped to 422 ("your money was refunded") — added a Redis `SET NX EX` Idempotency-Key so client retries can't double-charge, and proved it on real infrastructure: a test that kills the credit leg and asserts the source balance is restored *and* the ledger nets to zero, plus a mutation run (delete the refund → the test fails, showing money lost). **R:** payments stayed consistent under injected failures and retry storms, and the failure-handling design — not the happy path — became the part of the system we could walk auditors through line by line.

## Step 22 — Caching, Async & Clustered Scheduling (the Market-Info Read Model)

1. 🌟 **"How does `@Cacheable` work / what's cache-aside?"** *(the most commonly asked)* — A proxy intercepts the call, computes the key (SpEL over the arguments), and checks the cache: **hit** → return the cached value, the method body never runs; **miss** → run the body, store the result, return it. That's the app implementing **cache-aside** (look aside into the cache before/after loading from the source). `@CachePut` always runs and overwrites the entry (refresh); `@CacheEvict` removes. Strong answer adds the two ways this silently does nothing: no `@EnableCaching`, and self-invocation (`this.` never crosses the proxy).

2. **"What's a CQRS read model — and how is a cache one?"** — CQRS separates the write side from a **read-optimized view** that queries hit instead of the authoritative source. A TTL'd cache is the simplest read model: keyed for lookup, cheap to serve, decoupled from the source — and **eventually consistent** by design, with the TTL + refresh cadence as the documented staleness bound. (The full event-sourced projection version is the heavyweight sibling.) Framing staleness as a *product decision* rather than an accident is what makes this a senior answer.

3. **"Your `@Scheduled` job runs on 3 pods — what happens, and what do you do about it?"** *(gotcha favorite)* — `@Scheduled` is per-process, so the job runs 3× per tick: triple upstream load, duplicated work, possibly double-applied effects. Fix with a distributed lock per tick — **ShedLock**: before running, each instance tries an atomic claim on a named lock in a shared store (Redis `SET NX` + TTL — the same primitive as an idempotency key); only the winner runs, the rest skip *that tick* (no queueing, no retry). `lockAtMostFor` bounds a crashed holder (must exceed worst-case runtime — too short ⇒ overlapping runs); `lockAtLeastFor` floors the hold against clock skew/instant jobs.

4. **"When do virtual threads help — and how do you run `@Async` on them?"** — They help for **I/O-bound, mostly-waiting, high-concurrency** work: a virtual thread parks (unmounts from its carrier) instead of blocking an OS thread, so thousands are cheap. They don't speed up CPU-bound work. Wiring: provide a `SimpleAsyncTaskExecutor` with `setVirtualThreads(true)` as the `applicationTaskExecutor` bean — and note you **don't pool** virtual threads (pooling amortizes expensive threads; VTs are cheap-per-task, so create-per-task is the pattern). Bonus: make it falsifiable — return `Thread.currentThread().isVirtual()` from the async body and assert it in a test.

5. **"Why is a cached read eventually consistent, and how did you test that honestly?"** — Three sources of staleness: TTL (entries live until expiry), refresh cadence (updates land on the next tick), and **write-visibility** (a networked cache makes a just-written entry readable only after a round-trip). So the test must not assert instant read-after-write: we `await().untilAsserted(...)` that a repeat read is served from cache (provider call-count unchanged, value `equals`) — bounded polling that passes as fast as reality allows and fails loudly. A `Thread.sleep` would be the flaky/slow version of the same idea.

6. **"What's a cache stampede and how do you prevent it?"** — A hot key expires and every concurrent reader misses simultaneously → all of them hit the upstream at once (check-then-act across processes — a distributed race). Mitigations: keep hot keys warm with a scheduled refresher (pre-warming — our refresh job means tracked pairs essentially never miss cold), jittered early refresh, single-flight loading per key (`@Cacheable(sync = true)` does this *within one JVM only*), or serve-stale-while-revalidate.

7. **(Honesty/testing) "You can't spin up N pods in a unit test — how do you prove cluster-safe scheduling?"** — Prove the **mechanism** at the lock store: acquire the named lock, show a second acquire is *refused* while held, and re-acquirable after release — against real Redis (Testcontainers). Whether the second acquire comes from the same JVM or another pod is invisible to Redis, and that refusal is exactly what `@SchedulerLock` consults. Then observe the phenomenon by hand (two local instances, one "refreshed" log line per tick). Say explicitly which half is test-proven and which is observed — that distinction *is* the credibility.

> **Behavioral / STAR seed:** *"Tell me about a time you improved performance without compromising correctness."* — **S/T:** market-data reads hit a slow, rate-limited upstream on every request, and we were about to scale to multiple instances — which would also multiply our scheduled refresh by the instance count. **A:** I introduced a Redis-backed cache-aside read model with an explicit TTL (framed to the team as an eventually-consistent read model, so the staleness bound was a documented decision), moved warming off the request path onto virtual threads, and made the refresh cluster-safe with ShedLock over the Redis we already ran. I proved each property on real infrastructure: an instrumented (not mocked) upstream with delta assertions, an *awaited* write-visibility check instead of sleeps, a virtual-thread assertion, and a mutation run (delete `@Cacheable` → the test times out) showing the cache test had teeth. **R:** repeat reads dropped ~42ms → ~6ms measured, upstream volume became one fetch per pair per window regardless of instance count, and the suite demonstrably catches cache regressions.

## Step 23 — Onboarding Orchestration (Coordinating Services with Compensation)

1. 🌟 **"Orchestration vs choreography?"** *(the most commonly asked)* — Orchestration: a central coordinator calls each step and decides compensations — explicit, traceable, testable in one place; the coordinator is a coupling point. Choreography: services react to events with no coordinator — decoupled and extensible, but the flow is implicit across handlers and debugging means correlating event streams. Rule of thumb: orchestrate short critical flows you must reason about end-to-end (payments, onboarding); choreograph broad, loosely-coupled reactions (notifications, analytics). Strong answer: name one of each from a real system and *why* the style fit — ours: the Step-23 onboarding orchestrator vs the Step-20/21 event-driven notification path.

2. **"How do you undo a multi-service operation?"** — You can't: each step committed independently in its own service/DB, and commits don't un-happen. You run **compensation** — a new forward action that semantically undoes the effect (deactivate undoes create; refund undoes debit) — designed, coded, and *tested* as a first-class part of the flow. Volunteer the two fine points: compensation isn't isolation (intermediate states are visible until it runs), and compensation can itself fail (which is what pushes you toward durable/retryable workflows).

3. **"How does identity flow across services in a workflow?"** — For user-initiated flows, **token relay**: forward the caller's bearer token verbatim (`Authorization` header through the call chain); the downstream resource server validates signature/expiry/roles exactly as for a direct call. For service-initiated work (jobs, consumers), a **client-credentials** token — the service's own identity, no user context. The senior add-on: relay trades simplicity for full-token trust in the intermediary; token *exchange* (RFC 8693) narrows the downstream token when that matters.

4. **"Why timeouts on a coordinator's calls — and why two kinds?"** — A coordinator that blocks forever on one slow dependency holds a request thread hostage; under load that exhausts the pool and the outage cascades *upstream*. **Connect** timeout bounds "is the host reachable" (TCP handshake); **read** timeout bounds "is it answering" (time to response bytes). Both belong on the underlying client (JDK `HttpClient` connect / request-factory read) so every declarative call inherits them.

5. **(Gotcha) "Is your orchestrated workflow atomic or isolated?"** — Neither, and saying so confidently is the point. Not atomic: step 1 can commit and step 2 fail — compensation handles it. Not isolated: between steps, any reader sees the intermediate state (a customer without an account). You design for both: visible pending states, idempotent-shaped steps (our derived `DDA-<customerNumber>`; a set-to-constant deactivate), compensations safe to repeat.

6. **(Gotcha) "Your orchestrator crashed between create and compensate — damage and fix?"** — Damage: a committed customer with no account and no record of the in-flight workflow — invisible inconsistency; a synchronous in-memory orchestrator holds the workflow position in a call stack. Fixes, in escalating maturity: a reconciliation sweeper (find customers with no account older than N minutes), persisted saga state written *before* each step so a restart resumes/compensates, or a workflow engine (Temporal/Camunda) that makes the state machine durable by construction.

7. **(Testing) "How do you test a multi-service workflow without spinning up the services?"** — Over **real HTTP against in-process stubs**: boot the orchestrator on a random port, `@Import` a stub controller serving the downstreams' paths *in the same app*, build the clients with the same factory as production pointed at `localhost:<port>`, and make the stub failable on command (a real 500 body). You exercise the full client stack (JDK-proxy → RestClient → TCP → MVC) deterministically in seconds; then prove the test has teeth with a mutation run — delete the compensating call and the test fails (`Expecting actual: [] to contain exactly: [42]`). Keep the live multi-service check as a documented manual path and say which evidence covers what.

> **Behavioral / STAR seed:** *"Tell me about a time you designed for failure rather than just the happy path."* — **S/T:** retail onboarding spanned two services with separate databases — no shared transaction — and the bad outcome was a customer created without an account, invisible until support found it. **A:** I built a thin orchestrator with declarative HTTP clients (connect/read timeouts so it fails fast) and made the failure path first-class: on account-open failure it runs a compensating deactivate against the customer service and surfaces a typed 502 so callers know the system was left consistent; the caller's bearer token is relayed so the secured account service still authorizes the real user. I proved the failure path, not just the happy path: a real-HTTP test with failable in-process stubs asserts the exact compensation call, a mutation run (removing the compensation) makes it fail, and an ADR documents the accepted gap (synchronous, not crash-safe) with the path to durable workflows. **R:** a demonstrably self-cleaning workflow, a failure contract clients can program against, and a design review that took minutes because the trade-offs were already written down.

## Step 24 — Spring Batch (EOD Jobs) & the Phase-D Capstone

1. 🌟 **"What's a chunk-oriented step?"** *(the most commonly asked)* — Read items one at a time, process each (transform, filter with `null`, or reject), accumulate N, write the chunk — all inside one transaction, committed per chunk. N tunes throughput against memory and lock-hold time. Strong answer names the failure benefit: a crash loses at most one uncommitted chunk, and the JobRepository's bookmark (committed *with* each chunk) makes the rest resumable rather than re-runnable.

2. **"How do you stop one bad record from failing an EOD job?"** — Fault tolerance: `skip` that exception type up to a `skipLimit` (each occurrence recorded as a skip), `retry` transient types up to a `retryLimit`, route truly-bad records to a reject store for the morning team. The run completes; you reconcile the skips. Senior add-on: skip is an **allowlist of exception types** with a **bounded count** — unknown exceptions and mass failures should still fail the night, because 30,000 bad rows means the *feed* is broken, not the records.

3. **"What actually happens inside Spring Batch when a skippable exception is thrown mid-chunk?"** *(gotcha — separates readers from users)* — The chunk's transaction **rolls back** (the exception poisoned it), then Batch **re-processes the chunk's items one at a time in mini-transactions** (the *scan*) to isolate the offender; the bad item is recorded (`PROCESS_SKIP_COUNT`), survivors commit, `ROLLBACK_COUNT` rises. Corollary: skip is expensive — model expected no-ops as *filter* (processor returns `null`: no exception, no rollback, `FILTER_COUNT`), reserve skip for the exceptional.

4. **"What is the JobRepository and why version its schema with Flyway?"** — JDBC tables (`BATCH_JOB_INSTANCE/EXECUTION`, `BATCH_STEP_EXECUTION`, the `_CONTEXT` tables) recording every run's identity, attempts, counters, and restart bookmark. Version them like any schema — copy the canonical `schema-postgresql.sql` from your resolved spring-batch-core jar into a Flyway migration and set `spring.batch.jdbc.initialize-schema=never`: one schema owner, reviewable history, and no framework re-running `CREATE TABLE` against a production database at startup.

5. **"How does Batch restart work — and what stops a completed job from re-running?"** — Same **identifying** JobParameters → same `JobInstance`. A FAILED instance relaunches as a new `JobExecution` that resumes from the ExecutionContext's last-committed position (trustworthy because the metadata commits *with* the chunk, and the reader's order is deterministic — `id ASC`). A COMPLETED instance throws `JobInstanceAlreadyCompleteException` — the framework-level guard against double-crediting the book. That's why production EOD jobs use the **run date** as the identifying parameter.

6. 🌟 **(Capstone — the marquee)** **"How do you make a payment exactly-once across Kafka?"** — You can't make *delivery* exactly-once (acks can be lost; retries duplicate); you engineer exactly-once **effect**: at-least-once delivery (transactional Outbox so the event is never lost; the relay marks rows published only after the broker's ack) + an **idempotent consumer** deduping on an eventId minted once at the source and carried unchanged (outbox PK → Kafka record key → dedupe key). Our capstone proves it in one run: the same event delivered ≥ 2×, applied once. Depth: dedupe by *id*, never payload (two identical $40 payments are distinct events), and Kafka's own "exactly-once semantics" covers Kafka-to-Kafka topologies only — a database or HTTP call in the loop puts you back to idempotency.

7. **(Concurrency, Step-11/12 callback)** **"Your EOD job runs while live transfers are flowing — what goes wrong and how do you prevent it?"** — The lost update: the job read a balance, a live transfer changes it, the job writes interest computed against the stale read. Prevention at the database: the writer **re-reads each account `FOR UPDATE`** inside the chunk transaction (the same pessimistic lock as the safe transfer), so credits land on current state and concurrent writers serialize; transient conflicts are *retried* (`retry(OptimisticLockingFailureException.class)`), not skipped and not fatal. And the *trigger* needs cluster-safety too — ShedLock (or the JobInstance uniqueness as the second net), or three pods each run the night.

> **Behavioral / STAR seed:** *"Tell me about a time you built something that had to survive partial failure."* — **S/T:** our bank needed nightly interest accrual across every account, and the naive loop had two lethal failure modes: one malformed account aborting interest for everyone, and a crash mid-run forcing a full re-run that would double-credit completed accounts. **A:** I built it as a chunk-oriented Spring Batch job — committed chunks with the metadata bookmark in the same transaction (restart = resume, not redo), a skip allowlist with a bounded limit for per-record failures, retries for transient lock conflicts with live traffic, and row-level locks shared with the online transfer path. I versioned Batch's own metadata schema with Flyway like any other schema, and proved the fault tolerance with a mutation test: removing the skip config flips the job status COMPLETED→FAILED and the suite catches it. **R:** a night's run that demonstrably survives bad records (counted, reconciled, never silent), resumes after a crash without double-crediting, and a test suite that catches anyone who weakens the fault tolerance.

## Step 25 — SOLID & Clean Code (Refactoring a Smelly Service)

1. 🌟 **"Explain SOLID with a real example."** *(the most commonly asked)* — SRP: a Kafka listener that parsed JSON, deduped, built the message, and published had four-plus reasons to change; we extracted a parser, a dedup port, and a message factory, leaving a thin orchestration. DIP: the consumer now depends on a `ProcessedEventStore` interface it owns, not a `ConcurrentHashMap` — the store is swappable and mockable. Strong answers cite a real diff and name the concrete payoff: the extracted pieces gained unit tests that run in milliseconds with no broker and no Spring context.

2. **"What's a code smell — name a few and their fixes?"** — God method (one method, many jobs) → extract method/class along axes of change; feature envy (a method reaching into another structure's data — our `buildMessage(JsonNode)` with its `node.get(...).asText()` chains) → move the behaviour to the data's owner; tight coupling to a concrete type → depend on an interface (DIP); primitive obsession (a `JsonNode` standing in for a domain concept) → introduce a type (`TransferEvent` record). Smells are hints prompting a census, not rules demanding surgery.

3. **"What does Dependency Inversion buy you?"** — Three things: swappable implementations without touching callers (a Redis adapter becomes extension, not modification — OCP); isolated unit tests (the port can be mocked — even a lambda satisfies a one-method interface); and the correct dependency direction — policy and detail both point at an abstraction the *policy side owns*. The senior add-on: inversion is provable — `javap` the orchestrator's bytecode and the adapter's name appears nowhere.

4. **"How do you refactor safely?"** — Pin a green baseline first; move in small, committed, behaviour-preserving steps; run between moves; and **never change tests and production code together** — the unchanged suite is the judge. Our proof artifact: the three pre-existing integration test classes have a zero-line diff and still pass on a real broker. If the code has no tests, write characterization tests (pin current behaviour, bugs included) before moving anything.

5. **(Gotcha) "When is an abstraction NOT worth it?"** — One stable implementation, no test-seam need, no module boundary → YAGNI; the interface is ceremony. Add it when a real second case or a testability pain appears. Our one-adapter port cleared that bar three ways (a known Redis swap, a unit-test seam, the coming hexagonal boundary) — and the justification went into an ADR so the next reviewer doesn't have to re-litigate it.

6. **(Gotcha) "Your refactor broke a test — what does that tell you?"** — One of two findings: the move wasn't behaviour-preserving (most likely — undo it and take a smaller step), or the test asserted *structure* rather than behaviour (a test smell — fix the test first, in its own commit, then refactor). Either way the forbidden move is "fixing" the test in the same commit as the refactor: that converts your proof of safety into a rubber stamp.

7. **(Concurrency, Step-11 callback) "Your dedup store is shared by concurrent consumer threads — what's the contract and how do you meet it?"** — The port's contract is an *atomic* check-and-record: `markIfNew` returns `true` exactly once per id, even under concurrent duplicate deliveries. The in-memory adapter meets it with `ConcurrentHashMap.newKeySet().add()` — an atomic check-and-insert with no check-then-act window. A `contains()`-then-`add()` implementation (or a plain `HashSet`) type-checks, passes single-threaded tests, and still double-notifies under a concurrent duplicate — an LSP violation only a real-broker test or jcstress exposes. A distributed adapter meets the same contract with Redis `SET NX EX` (server-side atomicity across instances).

> **Behavioral / STAR seed:** *"Tell me about a time you improved code quality without breaking anything."* — **S/T:** our event consumer had grown into a god-method — parse, dedupe, build, publish in one `@KafkaListener` — coupled to concrete Jackson and an inline concurrent set; testing any one concern required a real broker, and the class changed for four unrelated reasons. **A:** I pinned a green baseline run, then refactored in committed micro-moves: extracted a domain event record and a parser (preserving its throw-on-poison contract because dead-letter routing depended on it), inverted the idempotency dependency behind a one-method port with the existing mechanism as its adapter, and moved message wording to a factory — explicitly leaving the integration tests untouched as the judge. I added fast unit tests for the extracted pieces, proved they had teeth with a mutation check (sabotaged the dedup, watched the test fail, reverted), and recorded the one-adapter-port justification in an ADR. **R:** identical observable behaviour (zero-diff test suite, green on a real broker), per-concern unit tests running in milliseconds without Docker, and a seam that later absorbed a durable Redis store with zero consumer changes — plus a written record of why the abstraction earned its place.

## Step 26 — Hexagonal Architecture (Ports & Adapters) + DDD

1. 🌟 **"Explain hexagonal / ports-and-adapters architecture."** *(The most commonly asked architecture question — have the picture ready to draw.)* — A framework-free domain + application core; the application exposes **inbound ports** (use cases it offers) and depends on **outbound ports** (things it needs); adapters at the edges plug into those ports. Source dependencies point **inward**: adapters → application ports → domain; the domain depends on nothing. Infrastructure (Kafka, HTTP, DB) is a swappable detail the core never imports. Testable without infra; swap the transport with a new adapter, zero core edits.

2. **"Inbound vs outbound port — give me a real example."** — Inbound (driving): what the application *offers* — `NotifyOnTransfer.handle(TransferEvent)`, implemented by `NotificationService`, called by the Kafka listener (a driving adapter). Outbound (driven): what the application *needs* — `ProcessedEventStore.markIfNew` and `NotificationPublisher.publish`, declared by the application, implemented by the in-memory store and the SSE hub (driven adapters). Same word "port," **opposite** implementation direction — the in/out distinguishes who implements it.

3. **"Why keep the domain free of framework imports?"** — Business rules survive framework churn; the domain is testable in microseconds without a container; and the dependency direction is *correct* — infrastructure depends on the core, not vice-versa. And it's **falsifiable**: our domain's entire import surface is one `import java.math.BigDecimal;` line, and `javap | grep -c adapter` on the use case's class file returns 0.

4. **"When do you NOT add aggregates and repositories in DDD?"** — When the context is thin — a read/push model with no invariant-guarding consistency boundary. Our notification service is value objects (`TransferEvent`, `Notification` records) + an application service; no aggregates, because there is nothing to guard. DDD tactical patterns are applied *where the domain warrants them*, not ceremonially. The richer aggregates live in the money domain (demand-account with its `Account` entity and `FOR UPDATE` locks).

5. **(Gotcha) "Isn't one interface per dependency a lot of ceremony?"** — Use ports where you have a real seam: a swap you can name, a test that needs the boundary, or a module edge worth enforcing. Our three ports all cleared that bar — a known Redis store swap (`ProcessedEventStore`), a unit-testable core without Docker, and a coming ArchUnit-enforceable boundary. Don't wrap every class; an interface with one implementation and no seam-need is YAGNI. The rule: add the abstraction when a concrete reason exists, and record the reason in an ADR so the next reviewer doesn't re-litigate it.

6. **(Gotcha) "How do you prove a restructure (a package move) was behaviour-preserving?"** — Three-step proof: (1) pin a **green baseline** before touching anything; (2) make the move structure-only — the pre-existing tests' diff must be **import lines only**, nothing else; (3) run the suite — unchanged assertions + green = the structural move didn't alter behaviour. Then add a **mutation check** to prove the new layer is on the hot path: we disabled `NotificationService`'s dedup guard and watched the unchanged `TransferEventConsumerKafkaTest` fail with `expected: 2 but was: 3`. Reverted; green. The new layer is exercised.

7. **(Concurrency, Step-11 callback) "Which threads run through your hexagon, and where are the concurrency guards?"** — Kafka listener threads drive `handle` while MVC request threads subscribe (`register`) and read (`recent`). Guards at the right rings: `ConcurrentHashMap.newKeySet().add()` atomically behind the `ProcessedEventStore` port (one check-and-insert, no check-then-act window); `CopyOnWriteArrayList` for safe concurrent iteration over live emitters in the SSE push adapter; `synchronized` bounded deque for the recent buffer; `final` fields throughout for safe publication to every thread that touches the beans. The use case itself is stateless — the cleanest concurrency property of all.

> **Behavioral / STAR seed:** *"Tell me about a time you paid down architectural debt without stopping feature work."* — **S/T:** our notification consumer had accumulated Kafka, SSE, DLT, and dedup features across three steps in a flat package; a previous SOLID pass had extracted collaborators but the push transport was still a concrete dependency, and a coming ArchUnit initiative needed real package boundaries to enforce. **A:** I proposed a behaviour-preserving hexagonal restructure scoped to one service: pinned the green baseline, introduced domain/application/adapter rings with explicit inbound and outbound ports, rewrote only the listener (to drive the inbound port rather than manage policy itself), and kept every pre-existing test's assertions untouched as the proof contract. I documented the two judgment calls (a `@Service` stereotype on the use case; one shared-SSE adapter coupling) in an ADR instead of hiding them, and verified the new layer was on the hot path with a mutation check. **R:** identical observable behaviour (test diff: import lines only, green on a real broker), a core unit-testable without Docker, a one-line domain-purity check (`grep "^import"` — one `java.math` line), and package boundaries the next step's ArchUnit rules enforced verbatim. The restructure was one step, not a rewrite quarter.

## Step 27 — Enforcing Architecture: ArchUnit + Spring Modulith (Fitness Functions)

1. 🌟 **"What's an architecture fitness function?"** *(Increasingly common as 'evolutionary architecture' enters interviews.)* — An automated test that asserts an architectural *characteristic* — here, structural dependency rules — so the design can't erode silently: a violating commit fails the build, not just a code review. Term from *Building Evolutionary Architectures*. It's the same shift as schema-as-migrations: the rule moves from a wiki diagram (which drifts) into the test suite (which is verified every run). We wrote four ArchUnit rules for the notification hexagon and a Spring Modulith `verify()` for demand-account's modules.

2. **"Does ArchUnit analyse source or bytecode — and why does it matter?"** — Bytecode: it imports the compiled `.class` files. The catch is that an **unused `import` is erased by the compiler**, so ArchUnit can't see it — only a *real* reference (annotation, field type, parameter, method call, supertype) is a dependency. That's the correct semantics (an unused import isn't a dependency) but it surprises people: to make a rule fail you must actually *use* the forbidden type. It's exactly why our §12.3 proof annotates the domain with `@Component` rather than just adding an import.

3. **"How does Spring Modulith decide what a module is, and what does `verify()` check?"** — Each **direct sub-package** of the application's base package is a module (demand-account → 9: batch, client, domain, event, outbox, payment, service, web, webhook). By default all of a module's types are API; types under an `internal` sub-package are hidden. `ApplicationModules.of(App.class).verify()` checks the two universal modular-monolith rules: **no cyclic dependencies** between modules, and **no access to another module's internals**. It's static bytecode analysis (ArchUnit under the hood) — no Spring context, no Docker.

4. **"ArchUnit vs Spring Modulith — when would you reach for each?"** — ArchUnit when you author *bespoke* rules about a specific design: a hexagon's layer rules, "controllers live in `..web..`", "no field injection". Spring Modulith for a *derived* module model plus the universal rules (cycles, internal access), living docs (`Documenter`), and optionally runtime module events/observability. They overlap (Modulith uses ArchUnit internally) but ArchUnit = custom rules, Modulith = module hygiene + docs. We used ArchUnit on the one hexagon and Modulith on the nine-module service.

5. **(Gotcha) "Your hexagon allows web→push SSE; how do the layered rules not flag that coupling?"** — We model **Adapter as a single layer**, so a web-adapter → push-adapter edge is *within* that layer and allowed; the rules still forbid any adapter → application/domain-inward dependency. If we'd split `adapter/in` and `adapter/out` into separate layers, the legitimate shared-SSE coupling (documented in ADR-0017) would fail. The lesson: model layers to match the *real* allowed edges, then enforce the edge that actually matters.

6. **"Why did you add Spring Modulith at test scope instead of on the main classpath?"** — Because this step wants *verification + docs*, not runtime change. `verify()` and `Documenter` are static analysis run in a test, so `spring-modulith-starter-test` + `spring-modulith-docs` at test scope suffice. Putting `spring-modulith-starter-core` on main would activate runtime module features (`ApplicationModuleListener`, module-scoped observability, externalized events) and startup cost — a behaviour change we deliberately deferred (ADR-0018 §3, "revisit if the capstone wants runtime module features").

7. **(Applied) "How do you prove a fitness function actually works — and why bother?"** — A guard you've never seen fail is worthless; it might be vacuous (e.g. `noClasses()` over an empty import passes trivially). So you make the architecture fail on purpose (a §12.3 mutation): annotate the pure domain with `@Component` → ArchUnit's `domain_is_framework_free` goes red, naming the class; add an `event→outbox` reference where `outbox→event` already exists → Modulith reports `Cycle detected` with the exact references on each leg. Revert → green. The red proves the rule is on the hot path; the green-after-revert proves you didn't leave the code broken.

> **Behavioral / STAR seed:** *"Tell me about a time you stopped a team's architecture from eroding."* — **S/T:** I'd just restructured a service into a hexagon (ADR-0017), but the boundaries were a *convention* — the next careless `@Component` on the domain or `import` from the core into an adapter would compile, pass review, and silently re-couple it. **A:** I made the boundaries executable: four ArchUnit rules encoding the hexagon's dependency rule over bytecode (modelling Adapter as one ring so a documented intra-adapter coupling stayed legal), plus Spring Modulith `verify()` on the richer nine-module service for cycle detection and living C4 docs. I kept Modulith at test scope to avoid touching runtime, and I *proved* each guard by injecting a real violation and watching the build go red before reverting. **R:** architecture became a build-failing test; the C4 diagram now regenerates from the verified model on every run, so docs can't drift; reviewers stopped having to remember the rule, and the next step's quality gates layered on cleanly. One test class per service, not a governance committee.

## Step 9 — Hibernate Performance & Correctness

1. 🌟 **"What is the N+1 SELECT problem, and how do you detect and fix it?"** *(most commonly asked)* — **N+1** is when loading a list of N parents fires **1** query for the parents and then **1 query per parent** to lazily load a child association — `1 + N` queries where you wanted `1`. It's silent because the code looks innocent (a loop reading a lazy collection). **Detect** it with Hibernate statistics (`hibernate.generate_statistics=true` → `getPrepareStatementCount()`), `spring.jpa.show-sql=true`, or a tool like datasource-proxy. **Fix** it with a fetch join (`JOIN FETCH`) or `@EntityGraph(attributePaths=…)` to load the children in one join — or, if you only need a few fields, a **projection** that avoids loading the association at all. Keep associations LAZY by default and opt into eager fetching per-query. For wide reads, **batch fetching** (`@BatchSize` / `default_batch_fetch_size`) is the middle ground.

2. 🌟 **"Optimistic vs pessimistic locking — when do you use each?"** *(concurrency)* — **Optimistic** (`@Version`): no DB locks during read/think-time; detect a conflict at write via `WHERE version=?` matching 0 rows → throw. Best when **conflicts are rare** (most edit screens) — cheap, scales well, but the loser must retry. **Pessimistic** (`SELECT … FOR UPDATE`, `LockModeType.PESSIMISTIC_WRITE`, Step 12): take a real row lock at read time so no one else can write until you commit. Best when **contention is high** or you must serialize (debiting a hot account) — correct under load but holds locks (less concurrency) and can deadlock. Both prevent **lost updates**; they differ in the bet on conflict frequency. In a bank: optimistic for customer/profile edits; pessimistic (or a serializable transaction / SELECT … FOR UPDATE) for ledger entries.

3. **"Why does `LazyInitializationException` happen, and why did turning OSIV *off* make it appear?"** *(version-evolution)* — A LAZY association is a proxy that loads only when touched **within an open Hibernate session**. If you touch it after the session/transaction closed, there's no session to run the query → `LazyInitializationException: could not initialize proxy - no Session`. **OSIV (Open-Session-in-View)** was historically **on by default** in Spring Boot: it kept a session open for the whole HTTP request (through view/serialization), so lazy loads "worked" even outside the service transaction — but that *hid* N+1s (queries firing during JSON serialization) and tied DB connections to request duration. We set `open-in-view: false`, so lazy-outside-transaction now **fails fast**, forcing you to fetch deliberately (`@EntityGraph`/projection) at the service layer. Modern senior default: OSIV off, fetch explicitly. (Boot still defaults it on and logs a warning.)

4. **"What exactly is the persistence context, and what's dirty checking?"** *(gotcha)* — The **persistence context** is Hibernate's per-transaction **1st-level cache**: a map of id → managed entity. Load the same id twice → one SQL `SELECT`, same instance returned. **Dirty checking**: Hibernate snapshots each managed entity on load and, at **flush**, compares current state to the snapshot and auto-generates `UPDATE`s for what changed — you never write `UPDATE` for a managed entity, just call a setter. **Flush** (push pending SQL to the DB) happens on commit, before an affecting query (AUTO mode), or on explicit `flush()` — and **flush ≠ commit** (flushed changes still roll back if the tx aborts). In tests, `clear()` detaches everything so subsequent reads hit the DB instead of the cache.

5. **"How does `@Version` prevent a lost update at the SQL level?"** *(applied)* — Hibernate adds the version to every `UPDATE`: `UPDATE customer SET ?, version = version + 1 WHERE id = ? AND version = ?` (the old version). The DB reports the affected-row count. If another transaction already incremented the version, the `WHERE` matches **0 rows** → Hibernate throws `StaleObjectStateException` → Spring's `ObjectOptimisticLockingFailureException`. No row was locked; the conflict is detected purely from the version number diverging. The losing transaction can then re-read and retry. That single extra `AND version = ?` predicate is the entire lost-update defence.

6. **"Interface vs class (DTO) projection — what's the difference and why use one?"** *(applied)* — A **projection** returns a subset of fields instead of full entities. An **interface projection** is a Java interface with getters; Spring Data backs it with a proxy and (for *closed* projections — all accessors map to properties) restricts the `SELECT` to just those columns. A **class/DTO projection** is a concrete class whose constructor parameters Spring Data binds. Use projections to (a) cut data over the wire, (b) avoid hydrating full entities and their lazy associations, and (c) practice **data minimization** — you can't leak a sensitive column you never selected. Use the full entity only when you need to *modify* it.

> **Behavioral / STAR seed:** *"Tell me about a time you diagnosed and resolved a database performance bottleneck."* — **S/T:** a page listing customers was getting slower as data grew; enabled SQL logging and found a classic N+1 query explosion where reading lazy addresses generated 1+N database roundtrips. **A:** wrote an integration test against a real Testcontainers PostgreSQL and counted statements using Hibernate's `Statistics` API (verifying 3 statement preparations for 2 seeded customers). Added a dedicated `@EntityGraph` query to fetch customers and addresses in a single SQL left join. **R:** verified statement count dropped from 3 to 1 in the test, eliminating the N+1 risk and avoiding connection pool starvation under load.
